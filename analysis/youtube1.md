## 1. Auto (`ooo auto`) — 이번 릴리즈(0.40.0) 핵심

- **철학: 인터뷰의 정반대.** 인터뷰 = 플라톤 동굴 비유. 결과물을 모른 채 벽에 비친 **그림자만 보고 원래 형체를 추론**. 근본 한계: 처음 질문을 제대로 못 만들면 온톨로지를 넓혀도 도달 어려움.
- **Auto = 꼬리.** 일단 던져서 결과를 뽑고, 그 **결과(꼬리)로 다시 머리(인터뷰)를 추론** → 사이클 반복. 우로보로스(자기 꼬리 무는 뱀) 이름값. "복잡계를 결과로서 다시 풀어나가는" 방식.
- **인터뷰는 1세션(사람 뇌 용량 한계), Auto는 10~20세션 병렬.** 프롬프트 복사해 최대 발산 → 텍스트/HTML 시안 10~20개 뽑음 → "몰랐던 것 발견" → 재인터뷰. 바이브코딩 시안 보고 다시 대화하는 흐름과 동형.
- **결정론적으로 모호도 0.2까지 깎음.** 단, LLM끼리 시뮬레이션하면 **plausible(그럴듯한) 답만 내서 0.2 밑으로 안 내려가는 경우 많음 — 의도된 동작.** 그게 곧 "시작 아이디어가 너무 모호했다"는 증빙. 루프 반복 시 **human-in-the-loop 강제 활성화**가 적절.
- **오실레이션 감지 → 서브에이전트 소환.** 원래 lateral-think(unstuck) MCP 툴 콜이던 걸 **서브에이전트로 전환**. 이유: 사람도 Claude/Codex도 서브에이전트를 "잘 본다"(Codex 앱의 레고 모양). 점수 안 내려갈 때 자기들끼리 논의시키니 잘 풀림.
- **목표 달성 방법 중 결과가 제일 빠름.** 개발자 자평.
- **가시성이 중요.** 화면 렌더링으로 진화 과정이 보이면 안심. 뒷단에서만 도는 백그라운드 작업은 불안 유발.

## 2. 인터뷰

- **모호도 ≤ 0.2까지 안 멈춤.** 문장 하나 → 인터뷰 → 결정론적으로 잡힌 것만 실행에 넘김.
- **"성과" 같은 단어의 다의성 해소가 목적.** 누구는 매출, 누구는 KPI. 인터뷰가 이걸 못박음.
- **탑다운 학습 도구로 인기.** AI는 보편 답변, 인터뷰는 심층 질문 → 모르는 용어가 나오면 그걸 공부하며 위에서 아래로 학습. "책 읽고 써먹기(바텀업)"의 반대.
- **"머리가 깔끔해진다"는 경험적 효과.** 같은 스펙 작업인데 OMC는 "머리에 남는 게 없다", 우로보로스는 정리되고 비워진 느낌. 기능이 아니라 체감.
- **사람이 병목.** 인터뷰에서 사람 생각이 병목이고, 사람은 시간 지나면 자기 생각을 까먹음 → **메모리 시스템 연결로 해소** 시도 (아래 7번).
- **다가올 업데이트:** 인터뷰 안에서도 forum research / 메인세션 자가답변 상황에 **서브에이전트 적극 활용** → UX 개선.
- **한계:** MCP로 던지는 인터뷰 에이전트는 **툴을 못 씀(답변만 가능)**. 툴 몇 개 주면 답 안 하고 툴만 쓰려 함. → 질문 전용 모델 파인튜닝 구상(질문 7 : 툴콜링 2 데이터셋).

## 3. 우로보로스는 원래 "제너럴 문제" 해결용 (코딩 아님)

- 지금은 거의 코딩에 쓰이지만 **범용 문제 해결 하네스로 설계됨.**
- 비개발 활용 사례: 교수의 논문 아이데이션, AI vs 사람 글 블라인드 테스트(프롬프트 A/B로 "AI 슬롭" 제거 패턴 찾기), 비행기표 구매(스카이스캐너·전 항공사 병렬 검색 → 최저가 → 결제는 grant/approver 승인), 밋업.
- **근석의 KTX/SRT 좌석예약 스킬 (실전 팁의 정수):**
  - caseskill 기본 KTX 예약은 빈약 — 아무 자리나 잡고 좌석 지정 안 됨. 원인: **좌석 서치 기능 자체가 없었음.**
  - 소비자 관점으로 재설계: 좌석 검색 API 추가 → **콘센트 있는 좌석 우선** → **정방향 우선** → **가운데 호차부터 탐색**(양 끝은 많이 걸어야 함). "변태적인 PR"이라 부를 만큼 디테일.
  - 이후 네이버지도+KTX+SRT+시외/고속버스 결합 최적 경로 탐색까지 확장.

## 4. 개발/아키텍처 판단 (재사용 가능한 교훈)

- **MCP 추상화가 최고의 결정.** 우로보로스는 자체 런타임이 있지만 **MCP로 추상화** → Codex 앱에 그대로 녹음. Teamworks는 Claude 전용이라 Codex 못 켬. 전사(비개발자 포함) Codex 이전이 이 덕에 수월.
- **스킬은 스크립트/워크플로우로 돌린다.** 근석 좌석 스킬은 "스킬 줄 수"가 아니라 **하드코딩 함수 + 알고리즘 = 워크플로우**. 약 1시간 작성, **토큰 소모 거의 없음.** → 결정론 로직은 LLM에 태우지 말고 스크립트로.
- **스킬은 소비자 관점으로 극단적으로 깎아라(F하게).** 버티컬/펫 스킬이 실사용을 만든다.
- **유비쿼터스 언어(DDD) 문제.** 여러 런타임에서 클라이언트마다 호출 프리픽스가 다름(`/goal` 슬래시, `$` 달러 등). 버그: 언어 설정 auto-detect라 영어면 `/goal` 동작, 한글이면 `목표`라야 동작 → **조용한 실패.** 해법 방향: `ooo` 키워드로 통일 매핑 + 런타임별 튜닝.
- **CLI → 앱 이동에 대비.** 앱은 골 실행 시 화살표 UI가 뜨고 진행도가 보여 **사람에게 안정감**을 줌. 플러그인 트리거 UI를 명시적으로 제공하는 게 사용성에 큼.
- **`/goal` + `ooo run` 조합이 좋다.** 60시간+ 자율 코딩 사례. Codex가 버그("빵꾸")를 기억해뒀다가 나중에 병렬 처리로 해결. Auto가 이 goal+run 조합만큼 좋아지는 게 목표.

## 5. Dynamic Workflow (Claude 신기능)에 대한 팀 견해 — 사용 시 기억할 것

- **토큰을 엄청 먹음.** 1M 위클리 리밋을 한 번에 20% 채움. 크리틱/리뷰 에이전트를 왕창 띄우고 왔다갔다 → 간단한 작업엔 토큰 아까움. **"싸지면 쓰겠다."**
- **고점(상한)을 뚫는 효과는 인정.** 방법 자체가 틀린 게 아니라 **연산 자원 가성비 문제.**
- **간단 작업은 스스로 거부함.** Claude가 "이 작업엔 적합한 태스크가 아니다"라며 실행 안 함(RLHF). 우로보로스도 "인터뷰할 게 아니다"라고 거부 — **좋은 시그널.**
- **서브에이전트 컨텍스트 불투명.** 다이나믹 워크플로우는 각 서브에이전트에 뭐가 넘어가는지 모름(Claude가 랜덤리 배분). 서브에이전트는 **메인 세션 안에 살아서 메인의 편향("바이러스")을 그대로 상속.** 트레이드오프: **롤을 완전히 정하면 전체 컨텍스트를 못 넘기고, 컨텍스트를 다 넘기려면 롤을 빼야 함.**
- **독립 세션 대안.** 자체 런타임으로 멀티세션을 돌리면 **capability graph로 툴을 제한**하고 넘길 컨텍스트를 정제 가능 → 우로보로스가 지향하는 방향.
- **반론(바이브코더 관점):** 어차피 코드 안 읽음. 잘 돌면 됨. 설계(목표+태스크+루프)를 앞단에서 잘 잡아두면 토큰값이 합당하고, 던져놓고 다른 계정으로 딴 작업하면 됨.
- **선례:** oh-my-codex "team mode"(코덱스 섹션 4~5개 자동 분할, 사용자에게 안 보임)와 유사. "와우 모먼트 + 뭐 하는지 모르겠음" 동시.

## 6. Codex 앱 (우로보로스 구동 환경)

- **우로보로스가 Codex에서 진짜 잘 돌아감** (MCP 기반이라). 팀 전사 Codex 이전 예정.
- 스레드 기능이 **비개발자에게 특히 좋음.** **PC↔PC 연결**(회사 컴 ↔ 집 컴)이 큰 변화. 맥미니=프론티어 모델, 맥북=로컬 모델 프로필 분리 활용.
- **"스레드 지우지 마라"** — Codex가 스레드/에이전트를 알아서 관리. 우로보로스도 효율적 스레드/세션 관리를 도입할 필요.
- **약점:** 디자인 결과물은 아직 Claude가 우위(Claude가 디자인 고집이 셈). 한글 뉴라인 깨짐(단어 중간 줄바꿈) → 별도 스킬 필요. Claude가 "박는다/척추" 같은 인터넷/개발자 슬랭을 과용(스크래핑 학습 추정).

## 7. 메모리 시스템 — 향후 최대 과제

- **Auto × 개인 메모리 시스템 연결이 핵심 태스크.** Obsidian 지식베이스, InfraNodus(온톨로지·그래프RAG)와 연결 → 인터뷰 질문에 **사람 대신 메모리에서 답을 찾아 회신** → 고도 자동화. (에르메스/Hermes 런타임에선 길어지면 에러 나던 이슈 있었음.)
- **"메모리는 증거가 아니라 베이지안 prior다."** 메모리를 증거로 쓰지 말고, **메모리 기반으로 증거를 찾아와라.** 안 그러면 에이전트가 숏컷을 탐.
- **벤치마크(논문 준비 중):** 메모리 유/무 = "툴콜 5번 안에 정답 찾나". 무메모리 10번 중 1번 정답. 유메모리 10번 중 10번 — 단 **3번은 5번 내 못 찾을 것 같으니 메모리로만 답함 = 30% 할루시네이션.** → 단기/장기 분리 + 장기 최소화 + **빠른 리콜용 인덱스(DB 인덱스처럼)** 필요.
- **"Wow 메커니즘" 논문(구글 딥마인드).** 트랜스포머에 외부 브레인 부착, 임팩트 큰 기억이 잘 남는 걸 수학적으로. → 봇도 필요. 임팩트 클 때 **세타(theta) 프로필(장기 추상화)이 크게 바뀌어야** 함.
- **Agent OS의 두 축:** (1) 메모리 리콜 + 장기 컨솔리데이션, (2) 에이전트 작업의 **결정론적 검증**("진짜 파일 수정했나"). Hermes가 스킬을 툴콜 로그로 검증하듯. 메모리 프로바이더가 스킬보다 2급 취급받는 게 문제. **Hermes식 무한 접지보다 결정론 게이트가 시간·토큰 절약.**
- **유저 스코어 기반 외부 RLHF 아이디어:** 작업 종료 후 사용자가 0~5점 → **가중치로 장기 메모리 승격 여부 결정.** 우로보로스는 SQLite DB 보유 → 구현 가능. 현재는 전부 단기 넣고 시간 따라 리랭크 하락 방식 → 명시적 유저 스코어가 나음. TUI/uro-code에서 세션 그룹 리스트업 후 평가.
- **파인튜닝:** Thinker/Atropos 스킬(Thinking Machines·Mira Murati / Nous·Hermes·Teknium 계열)로 LoRA·모델 최적화를 쉽게. 우로보로스 자체 모델을 유저 메모리로 깎으면 유저 선호로 가중, 저점수는 진행 안 함. 모델+메모리 강화학습까지 가면 하네스로 "넘사벽".

## 8. Agent OS — 1.0 로드맵

- **0.4 → 1.0 = "Agent OS".** 기존 우로보로스가 "프롬프트"에 관한 것이었다면, Agent OS는 **여러 에이전트를 포괄하는 OS/커널이 되겠다는 선언.** "어떤 오픈소스가 나와도 우리 생태계로 편입, 우로보로스 안에서 해결." 오픈소스를 보는 철학의 표현이라 릴리즈에 신중.
- **어댑터:** Pi 커스텀 하네스(승현의 Notch Pi), goose CLI 등 **9~10개.** 어떤 하네스든 우로보로스 계약(contract) 안에서 돌게 함.
- **1.0 조건:** 플러그인 생태계 + **uro-code**(OS 커널을 보여줄 클라이언트/유저 레벨 프로그램). 6월 목표.
- **일정:** 6월 1.0 → 플러그인 다수 제작 → **MCP 브레이킹 체인지(7/28~30, 헤더 기반 인증 추가)에 맞춰 7월 말 2.x** — 우로보로스를 네트워크에 올려 표 구매 등 실물 문제 해결.
- **Seed 온톨로지 필드 개선 아이디어:** 유저의 닫힌 계(메모리 프로바이더) 안에서 bounds/action/property를 **온톨로지틱하게 정의** → 단어가 그래프/데이터 스키마로 결정론 매핑. InfraNodus MCP가 이 기능 제공. **"온톨로지가 자동화의 핵심 — 없으면 계속 슬롭만 뱉음."**

## 9. AI 개발자가 기억할 것 (요약)

1. **결과 → 재인터뷰 루프.** 모호하면 인터뷰로 완벽히 못 잡음. **일단 던져(Auto) 결과를 보고 다시 질문**하는 게 빠르다. 0.2 밑으로 안 내려가면 그건 "아이디어가 모호하다"는 신호.
2. **결정론 로직은 LLM 말고 스크립트/워크플로우로.** 토큰 안 먹고 재현성 확보(근석 좌석 스킬 = 1시간·거의 0토큰).
3. **스킬은 소비자 관점으로 극단적으로 깎아라.** 버티컬/펫 스킬이 실사용을 만든다.
4. **다이나믹 워크플로우 = 고점은 높지만 토큰 가성비 낮음.** 간단 작업엔 쓰지 마라(에이전트도 거부함). 서브에이전트에 뭐가 넘어가는지 불투명함을 인지.
5. **서브에이전트는 메인 세션의 편향을 상속.** 롤 vs 컨텍스트는 트레이드오프. 격리가 필요하면 독립 런타임 + capability graph로 툴 제한.
6. **MCP 추상화 = 이식성.** 특정 앱(Claude) 전용으로 묶지 말면 Codex 등 어디서든 돈다.
7. **런타임/언어별 명령어 매핑 조심.** auto-detect 언어 설정이 조용한 실패를 만든다.
8. **메모리는 prior지 증거가 아니다.** 메모리로 답하면 할루시네이션(사례: 30%). 단기/장기 분리 + 인덱스 + 결정론 검증 게이트.
9. **작업 후 유저 스코어로 메모리 승격/RLHF.** 개인화의 값싼 지렛대.
10. **온톨로지가 자동화의 핵심.** 잘 구축돼 있으면 자동화, 없으면 슬롭.
