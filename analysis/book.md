# 1장. 입력의 실패 — Safe-but-Wrong의 세계

> "대부분의 AI 코딩 실패는 출력이 아니라 **입력**에서 발생한다. 병목은 AI의 능력이 아니라 인간의 명확성이다."
> — Ouroboros README

## 1.1 우리가 겪는 실패의 진짜 모양

AI 에이전트에게 일을 시켜본 사람이라면 누구나 이런 경험이 있습니다:

- "태스크 관리 앱 만들어줘" → 그럴듯한 코드가 나왔지만 내가 원한 것이 아니다
- 에이전트가 "완료했습니다!"라고 했지만 실행해 보니 절반만 동작한다
- 세 시간짜리 자율 실행이 끝났는데, 처음 요구사항에서 슬금슬금 벗어나 있다
- 테스트는 전부 통과하는데... 테스트가 검증하는 것이 내 요구사항이 아니다

이 실패들의 공통점은 무엇일까요? **에이전트가 멍청해서가 아닙니다.** 에이전트는 주어진 입력에 대해 대체로 합리적으로 행동했습니다. 문제는 입력 — 즉 우리가 전달한 의도 — 이 애초에 모호했고, 그 모호함이 어디서도 **측정되거나 차단되지 않은 채** 실행 단계까지 흘러들어 갔다는 것입니다.

## 1.2 가장 위험한 실패: Safe-but-Wrong

Ouroboros 문서 중 가장 통찰력 있는 글은 `docs/guides/safe-but-wrong-output.md`입니다. 이 문서는 에이전트 실패 중 **가장 잡기 어려운 부류**에 이름을 붙입니다:

> "실행 결과는 안전하고, 파괴적이지 않고, 심지어 유용하면서도, 사용자가 말한 목표에 대해서는 **틀릴** 수 있다."
> (*"can be safe, non-destructive, and even useful, while still being wrong for the user's stated goal."*)

핵심 속성은 이것입니다:

> "일반적인 안전 검사는 전부 통과할 수 있다. 파일은 삭제되지 않았고, 외부 시스템은 변경되지 않았고, 테스트는 통과할 수 있고, 출력은 읽을 만하다. 그런데도 틀렸다 — **산출물 계약(artifact contract)이 표류했기 때문이다.**"

그리고 왜 이것이 눈에 보이는 실패보다 나쁜지:

> "명백한 차단(blocker)보다 나쁘다. **거짓 확신(false confidence)을 만들기 때문이다.**"

### Safe-but-Wrong의 분류학

Ouroboros가 문서화한 실제 사례 유형들 — 여러분도 분명 겪어봤을 것입니다:

| 사용자가 요청한 것 | 에이전트가 만든 것 |
| :--- | :--- |
| 실행 가능한 도구 | 인수인계 문서 뭉치 |
| 재사용 가능한 워크플로 | 일회성 체크리스트 |
| 배치 처리 표면 | 단일 케이스 요약 |
| 결측 데이터의 정직한 표기 | `OK`, `0`, 빈 값으로 위장된 결측치 |

마지막 항목이 특히 교활합니다. "데이터가 없어서 확인 못 함"을 `0`이나 `OK`로 표기하면, 하류의 모든 판단이 오염됩니다. Ouroboros는 이를 명시적으로 금지합니다: 결측 입력은 `insufficient data` / `unchecked`로 표기해야 합니다.

### 보고 언어의 대비

같은 상황에 대한 두 가지 보고:

```
나쁨:  "Done. Created the handoff package."
좋음:  "Status: partial/supporting-output, not complete."
```

첫 번째는 실제로 한 일(문서 작성)을 완료 선언으로 포장합니다. 두 번째는 산출물의 **종류**가 요청과 다르다는 사실을 정직하게 드러냅니다. Agentic Coding에서 보고 언어의 정확성은 사치가 아니라 안전장치입니다 — 에이전트의 보고가 다음 판단의 입력이 되기 때문입니다.

## 1.3 방어: 산출물 계약 6점 검사

Ouroboros는 safe-but-wrong을 잡기 위해 산출물을 원래 계약과 대조하는 6가지 질문을 제시합니다:

1. **산출물의 종류(class)** — 사용자가 요청한 것과 같은 *종류*의 것인가? (도구 vs 문서)
2. **실행 표면** — "새 입력을 넣어 다시 실행할 수 있는가?"
3. **보조 산출물의 위장** — 지원 자료가 최종 산출물 행세를 하고 있지 않은가?
4. **결측 데이터 의미론** — 없는 것이 없다고 표기되어 있는가?
5. **과거 교정 이력** — 사용자가 이전에 바로잡은 사항이 다시 어겨지지 않았는가?
6. **검증 증거** — "검증이 *요청된 동작*을 증명했는가, 아니면 *어떤 파일이 존재한다는 것만* 증명했는가?"

6번이 백미입니다. "빌드가 성공했다"와 "요청한 기능이 동작한다"는 서로 다른 명제인데, 에이전트의 자기 보고에서는 자주 뒤섞입니다.

균형 감각도 중요합니다. 이 문서는 모든 임의 판단을 금지하지 않습니다:

> "보수적 기본값은 국소적이고 되돌릴 수 있는 공백을 메울 때는 괜찮다. **사용자가 받기로 한 것의 종류를 바꿀 때** 위험해진다."

에이전트가 사소한 결정(변수명, 파일 배치)을 스스로 내리는 것은 생산적입니다. 위험한 것은 산출물의 **본질**을 바꾸는 무단 결정입니다.

## 1.4 일반화: Agentic Coding의 제1 문제 설정

이 장에서 추출할 원칙:

**원칙 0 (문제 인식): 에이전트 실패의 기본형은 "능력 부족"이 아니라 "계약 표류"다.**

따라서 Agentic Coding 체계가 가장 먼저 갖춰야 할 것은 더 강한 모델이 아니라:

1. **계약을 명시하는 장치** — 무엇을 만들기로 했는지가 검사 가능한 형태로 존재해야 한다 (→ 2장)
2. **계약과 대조하는 장치** — "완료" 주장을 계약과 대조 검증해야 한다 (→ 3장)
3. **표류를 측정하는 장치** — 진행 중에 계약에서 얼마나 멀어졌는지 관측해야 한다 (→ 4장)

## 1.5 Claude Code 적용

이 장의 통찰을 오늘 바로 적용하는 방법:

### 산출물 종류를 먼저 못 박기

작업 요청 시 산출물의 **종류**를 명시하세요. Claude Code의 CLAUDE.md나 프롬프트에:

```markdown
## 산출물 계약
- 이 작업의 최종 산출물: 실행 가능한 CLI 도구 (문서 아님)
- 완료 기준: `mytool --input sample.csv`가 새 입력에 대해 재실행 가능해야 함
- 결측/미확인 항목은 반드시 "unchecked"로 표기 (OK/0으로 채우지 말 것)
```

### "Done" 검증 습관

에이전트가 완료를 선언하면 6점 검사 중 최소 2개를 물어보세요:

- "새 입력으로 다시 실행하는 명령어를 보여줘" (실행 표면)
- "검증이 증명한 것이 정확히 뭐야? 파일 존재? 아니면 요청한 동작?" (검증 증거)

### 보고 언어 규칙을 CLAUDE.md에

```markdown
## 보고 규칙
- 부분 완료는 "부분 완료"로 보고한다. 지원 산출물을 완료로 포장하지 않는다.
- 테스트 실패, 건너뛴 단계는 숨기지 말고 그대로 보고한다.
```

Claude Code는 CLAUDE.md의 지시를 시스템 프롬프트 수준으로 준수하므로, 보고 정직성 규칙은 여기에 두는 것이 가장 효과적입니다.

# 2장. 원칙 I: 명확해지기 전에는 만들지 마라

> "인터뷰는 당신이 준비되었다고 *느낄 때* 끝나지 않는다. **수학이 준비되었다고 말할 때** 끝난다."
> — Ouroboros README

## 2.1 모호함을 숫자로 만들기

1장의 진단이 옳다면 — 실패는 입력에서 시작된다 — 처방은 명백합니다: **입력의 품질을 측정하고, 기준 미달이면 실행을 거부하라.** 문제는 "모호함"이 주관적 느낌처럼 보인다는 것입니다. Ouroboros의 첫 번째 공학적 기여는 이것을 반박한 것입니다.

`src/ouroboros/bigbang/ambiguity.py`의 구현:

```
Ambiguity = 1 − Σ(clarity_i × weight_i)
```

각 명확도 차원은 LLM이 0.0~1.0으로 채점하되, **temperature 0.1** (`SCORING_TEMPERATURE = 0.1` — "재현 가능한 채점을 위해")로 고정합니다. 가중치는 프로젝트 유형에 따라 다릅니다:

| 차원 | 신규(Greenfield) | 기존 코드(Brownfield) | 묻는 것 |
| :--- | :---: | :---: | :--- |
| Goal Clarity | 0.40 | 0.35 | 목표가 구체적인가? |
| Constraint Clarity | 0.30 | 0.25 | 제약이 정의됐는가? |
| Success Criteria | 0.30 | 0.25 | 결과가 측정 가능한가? |
| Context Clarity | — | 0.15 | 기존 코드베이스를 이해했는가? |

**관문: `AMBIGUITY_THRESHOLD = 0.2`.** 이 값 아래로 내려가기 전에는 명세(Seed) 생성이 거부됩니다.

왜 0.2인가? 가중 명확도 80% 지점에서는 남은 미지수가 코드 레벨 결정으로 흡수 가능할 만큼 작기 때문입니다. 그 위에서는 아직 **아키텍처를 추측**하고 있는 것입니다.

세부 설계에도 배울 점이 많습니다:

- **차원별 하한선(completion floor)**: 전체 점수와 별개로 Goal ≥ 0.75, Constraint ≥ 0.65, Success ≥ 0.70을 각각 요구합니다. 한 차원의 우수함이 다른 차원의 결핍을 가리는 것(평균의 함정)을 막습니다.
- **의도적 유보는 감점하지 않음**: "나중에 결정" 항목은 *의도적*이므로 점수를 깎지 않습니다. 명확함이란 모든 것을 아는 것이 아니라 **모르는 것을 아는 것**입니다.
- **Brownfield는 차원이 하나 더**: 기존 코드베이스 작업에서는 "코드를 이해했는가"가 별도 채점 차원(15%)입니다. 코드 이해 없는 요구사항 명확화는 반쪽입니다.

## 2.2 질문하는 에이전트: Socratic Interviewer

점수를 낮추는 수단은 **질문**입니다. `agents/socratic-interviewer.md`의 페르소나 정의는 역할 경계가 인상적으로 엄격합니다:

> "당신은 오직 인터뷰어다. … 절대 'X를 구현하겠습니다', '만들어 보겠습니다'라고 말하지 않는다 — 당신은 요구사항만 수집한다."
> (*"You are ONLY an interviewer. … NEVER say 'I will implement X', 'Let me build', 'I'll create'"*)

> "당신은 질문 생성기다. 당신의 일: **모호함을 가장 크게 줄이는 단 하나의 최선의 소크라테스식 질문**을 생성하는 것."

주목할 프롬프트 설계 장치들:

- **최대 정보 이득**: 아무 질문이나 던지는 것이 아니라 "모호함의 가장 큰 원천"을 겨냥합니다. 질문에도 우선순위 함수가 있습니다.
- **폭 제어(breadth control)**: "하나의 파일·추상화·버그가 여러 라운드를 연속 지배했다면, 명시적으로 줌아웃하라." 대화가 국소 최적점에 빠지는 것을 프롬프트 수준에서 방지합니다.
- **출처 태깅**: Brownfield 인터뷰에서 모든 사실에 `[from-code]` / `[from-user]` / `[from-research]` 태그를 붙입니다. 그리고 "'무엇이 존재하는가'가 아니라 '왜?'와 '무엇이 바뀌어야 하는가?'를 물어라."

## 2.3 결정화: 불변 명세 (Seed)

관문을 통과하면 대화는 **Seed** — "워크플로의 헌법(constitution)" — 로 결정화됩니다. `core/seed.py`에서 Seed는 frozen Pydantic 모델입니다:

> "일단 생성되면 Seed는 수정될 수 없다."
> — `docs/architecture.md` 설계 원칙 #2: "**불변의 방향(Immutable Direction)** — Seed는 변하지 않는다; 그것을 달성하는 경로만 적응한다."

왜 불변성이 그토록 중요한가? **불변성은 강제 함수(forcing function)이기 때문입니다.** Seed를 바꾸는 유일한 경로는 다중 모델 합의 게이트(3장)를 통과하는 명시적 절차입니다. 실행 도중 에이전트가 "이게 더 나을 것 같은데"라며 조용히 목표를 재해석하는 일 — 1장의 계약 표류 — 이 **구조적으로 불가능**해집니다.

### 수용 기준의 입도(granularity) 계약

Seed를 만드는 `agents/seed-architect.md`에는 과분해를 막는 강력한 규칙이 있습니다:

> "수용 기준(AC)은 3~7개. 각각은 **독립적으로 가치 있고 사용자에게 보이는 결과** 하나다 — 구현 단계가 아니다."
> "형제 AC의 하위 단계인 AC는 … **결함이며, 누락된 요구사항과 같은 심각도다.**"
> "작업을 원자 단위로 쪼개는 것은 실행 엔진의 런타임 책임이다; 여기서 하면 아무 이득 없이 토큰 비용만 곱한다."

명세는 *무엇*의 목록이지 *어떻게*의 목록이 아닙니다. 명세 단계에서 구현을 분해하면 (1) 비용이 늘고 (2) 실행 엔진의 최적화 여지를 빼앗고 (3) 명세가 구현에 종속됩니다.

또 하나, 실패에서 배운 흔적이 역력한 형식 안전 규칙:

> "AC의 `verify`는 정확히 한 줄짜리 셸 명령. heredoc이나 멀티라인 셸 문법 절대 금지."
> "`expect`는 verify 명령이 stdout에 그대로 출력하는 리터럴 문자열만. 'exit code 0', 'success', 'passed' 절대 금지."

검증 명령의 모호함은 검증 자체를 모호하게 만듭니다. 기대값이 "success"라면 무엇이든 성공처럼 보이게 만들 수 있습니다(3장의 리워드 해킹과 연결).

### 승격 정책: 추측을 요구사항으로 만들지 마라

> "결정론적 요구사항 승격 정책(Requirement Promotion Policy)이 주어지면 그것이 권위다: 승격 목록에 있는 후보만 하드 요구사항이나 수용 기준이 될 수 있다. … 제품 레퍼런스, 용어 설명, 시각적 취향 신호, **모델의 추측**을 사용자의 명시적 확인 없이 수용 기준으로 바꾸지 마라."

LLM은 대화에서 온갖 것을 "요구사항 같은 것"으로 승격시키고 싶어 합니다. 이 규칙은 **요구사항의 혈통(provenance)** 을 지킵니다: 사용자가 말했거나 확인한 것만 계약이 됩니다.

## 2.4 합의 수렴의 반칙 방지

`ooo auto`의 수렴 계약(`docs/auto-interview-convergence-contract.md`)에는 자동화가 게으르게 수렴하는 것을 막는 조항이 있습니다:

> "인터뷰 단계는 **우연히 수렴해서는 안 된다.**"

모든 필수 섹션은 네 가지 중 하나로 귀결되어야 합니다: (1) 사용자 제공 사실, (2) 한정된 저장소 사실, (3) 안전한 자동 가정 — 단 "국소적이고, 되돌릴 수 있고, 파괴적이지 않아야" 하며 근거와 함께 출처 태깅, (4) 명시적 차단(blocker). 그리고:

> "진행한다는 것은 단지 `blocked`를 피했다는 뜻이 아니다: 결과 원장에는 **감사 가능한 가정, 비목표, 수용 기준, 검증 계획**이 포함되어야 한다."

차단 사유조차 정직해야 합니다: "최대 라운드 수에 도달했다"가 아니라 **어떤 안전하지 않은 결정이 미해결인지** 를 명시해야 합니다.

## 2.5 일반화: 원칙 I

**명확해지기 전에는 만들지 마라. 그리고 "명확함"을 측정 가능하게 정의하라.**

하위 원칙들:

1. **모호함은 측정 대상이다** — 느낌이 아니라 차원별 점수와 임계값으로.
2. **질문은 정보 이득 순으로** — 가장 큰 모호함부터 공략하고, 국소 최적에 빠지면 줌아웃.
3. **의도는 불변으로, 경로는 가변으로** — 목표·제약·수용 기준을 고정하면 표류가 구조적으로 차단된다.
4. **명세는 결과의 목록, 3~7개** — 구현 분해는 실행기의 몫.
5. **추측은 계약이 아니다** — 사용자가 확인한 것만 요구사항으로 승격.
6. **모르는 것은 모른다고 기록** — 의도적 유보는 결함이 아니라 명확함의 한 형태.

## 2.6 Claude Code 적용

### 계획 모드 = 나의 모호도 게이트

Claude Code의 **계획 모드(Plan Mode)** 는 이 원칙의 내장 구현입니다. 실행 전에 계획을 세우고 승인받는 관문이죠. Ouroboros식으로 강화하려면:

```markdown
# CLAUDE.md
## 착수 전 규칙
복잡한 작업은 코드 수정 전에 다음을 확인하고, 불명확하면 질문한다:
1. 목표: 사용자에게 보이는 결과가 무엇인가? (구체적으로)
2. 제약: 건드리면 안 되는 것, 지켜야 할 것은?
3. 성공 기준: 완료를 어떻게 검증하는가? (실행 가능한 명령으로)
세 가지 중 하나라도 추측에 의존하면, 추측을 명시하고 확인을 요청한다.
```

### 인터뷰를 먼저 시키기

큰 작업은 구현 요청 전에 인터뷰를 요청하세요:

```
이 기능을 바로 구현하지 말고, 먼저 요구사항을 명확히 하는 질문을 해줘.
한 번에 하나씩, 모호함이 가장 큰 부분부터. 내 답변이 충분히 명확해지면
수용 기준 3~7개로 된 명세를 작성해서 확인받은 뒤 시작해.
```

### 명세 파일 = 나의 Seed

승인된 계획을 파일로 고정하세요 (`SPEC.md`, 이슈, PR 본문 등):

```markdown
# SPEC: CSV 내보내기 기능
## 목표 (불변)
사용자가 필터링된 거래 내역을 CSV로 내보낼 수 있다.
## 제약 (불변)
- 기존 export 모듈 인터페이스 유지
- 스트리밍 처리 (전체 메모리 적재 금지)
## 수용 기준 (각각 검증 명령 포함)
1. 필터 적용 상태로 내보내기 — `pytest tests/test_export.py::test_filtered`
2. 10만 행에서 메모리 상한 유지 — `pytest tests/test_export.py::test_memory`
3. UTF-8 BOM 옵션 — `pytest tests/test_export.py::test_bom`
## 비목표
- Excel(.xlsx) 형식 (다음 이터레이션)
## 가정 (사용자 확인 필요)
- [가정] 구분자는 쉼표 고정 → 확인됨 2026-07-19
```

세션이 길어지거나 컨텍스트가 압축돼도 이 파일이 **의도의 앵커**가 됩니다. 에이전트가 표류하면 "SPEC.md와 대조해줘" 한 마디로 복귀시킬 수 있습니다.

# 3장. 원칙 II: 완료는 승인이 아니다

> "실행은 평가가 아니다. 완료는 승인이 아니다. 태스크 실패는 의미론적 표류가 아니다."
> (*"Execution is not evaluation. Completion is not approval. Task failure is not semantic drift."*)
> — `docs/guides/execution-vs-evaluation.md`

## 3.1 세 가지 질문을 뒤섞지 마라

에이전트가 "완료했습니다"라고 말할 때, 사실 세 개의 서로 다른 질문이 있습니다:

1. **작업자가 일을 끝냈는가?** — 실행 상태 (completed / failed / blocked / skipped)
2. **산출물이 수용 기준을 만족하는가?** — 공식 평결 (pass / fail / **not_evaluated**)
3. **의도에서 표류했는가?** — 드리프트 점수 (실제로 측정했을 때만 존재)

Ouroboros는 이 셋을 **타입 수준에서** 분리합니다. 하나의 수용 기준(AC)은 두 개의 런타임 산출물 — `TaskResult`와 `ACResult` — 을 갖고, 문서는 이렇게 못 박습니다:

> "완료된 태스크는 실행이 끝났다는 **증거**일 뿐이다. AC가 통과했다는 **증명**이 아니다."

가장 배울 만한 것은 **보수적 기본값**입니다:

> "검증자가 기대 AC의 일부만 다뤘다면, 다루지 않은 AC는 승인되지 않는다. 공식 평결이 존재할 때까지 `not_evaluated`로 남는다."

그리고 값싼 완료 비율로 "`drift_score`를 합성해서는 안 된다" — 측정하지 않은 것을 측정한 척하지 않습니다. 1장에서 본 결측 데이터 의미론(`0`이 아니라 `unchecked`)의 타입 시스템 버전입니다.

## 3.2 검증의 비용 사다리: $0 → $$ → $$$

검증이 중요하다고 해서 모든 검증에 프런티어 모델을 쓰면 파산합니다. Ouroboros의 평가 파이프라인(`evaluation/pipeline.py`)은 **비용 단조(cost-monotonic)** 구조입니다:

> "싼 검사가 비싼 검사의 관문이 된다: Stage 1은 무료, Stage 2는 LLM 호출 한 번, Stage 3(다중 모델 합의)는 특정 트리거가 있을 때만 실행된다."

### Stage 1 — Mechanical ($0)

린트, 빌드, 테스트, 정적 분석, 커버리지(임계 0.7). LLM 없음. **하나라도 실패하면 Stage 2·3은 아예 건너뜁니다.** 컴파일도 안 되는 코드의 의미론을 LLM에게 물어보는 것은 낭비입니다.

이 단계의 명령어 자동 탐지(`evaluation/detector.py`)는 방어적 엔지니어링의 교본입니다:

- LLM이 저장소 매니페스트를 보고 검사 명령을 **제안**하지만, 모든 제안은 디스크 위의 실제 진입점과 대조 검증됩니다: "Stage 1은 절대 유령 실패(phantom failure)를 만들어서는 안 된다."
- 셸 체이닝 토큰(`&&`, `|`, `;`, `>`, `` ` ``, `$(`) 전면 금지, 실행 파일 허용목록(allowlist), `npm` 스크립트는 package.json에 실제 존재해야 하고, `make` 타겟은 존재 + **비파괴**(deploy/publish/push/reset 등 거부)여야 합니다.
- 호스트 `PATH`는 **의도적으로** 조회하지 않습니다 — 개발자 머신에만 있는 전역 도구가 CI에서 유령 실패가 되면 안 되므로.
- 신뢰할 수 없는 저장소 설정이 임의 명령을 실행하지 못하도록, 프로젝트 오버라이드도 허용목록 안에서만 동작합니다.

**교훈: AI의 제안은 결정론적 검증기를 통과한 뒤에만 실행하라.**

### Stage 2 — Semantic ($$)

LLM 한 번이 구조화된 JSON을 반환합니다: `score`, `ac_compliance`, `goal_alignment`, `drift_score`, `uncertainty`, 그리고 `reward_hacking_risk`. 승인 조건: `ac_compliance == True AND score ≥ 0.8`, 트리거 미발동.

중요한 지원 장치 — **평가자에게 진짜 코드를 준다**:

> "이것은 의미론적 평가자에게 에이전트의 텍스트 요약이 아니라 **실제 코드**를 준다."

에이전트의 자기 보고("구현했습니다")가 아니라 산출물 자체(파일 30개, 파일당 50KB 한도)를 증거로 수집해 평가합니다. **에이전트의 주장을 평가하지 말고 에이전트의 산출물을 평가하라.**

### Stage 3 — Consensus ($$$)

트리거(Seed 수정 시도, 온톨로지 진화, 드리프트 > 0.3, Stage 2 불확실성 > 0.3 등)가 발동할 때만 다중 프런티어 모델이 투표합니다. **2/3 다수결**(`majority_threshold = 0.66`)로 승인. 심화 모드는 법정 구조입니다: **Advocate(변호인) + Devil's Advocate(악마의 변호인)** 가 1라운드에서 동시 변론하고, **Judge(판사)** 가 2라운드에서 `approved / rejected / conditional` 평결을 냅니다.

트리거 행렬은 우선순위 순서로 **첫 매치만 발동**합니다 — 여러 경보가 동시에 울릴 때의 혼선을 설계 단계에서 제거한 것입니다.

## 3.3 리워드 해킹: 평가를 속이는 산출물

가장 정교한 방어는 **평가 자체를 겨냥한 게이밍**에 대한 것입니다. Stage 2 프롬프트의 "Anti-Gaming Verification" 블록은 다음을 찾도록 지시합니다:

- 하드코딩된 출력
- 테스트에서만 타는 분기(test-only branches)
- 자리표시자(placeholder) 로직
- 명시적 예시에만 맞춘 좁은 구현

그리고 "실제 문제를 풀지 않고 점수만 잘 받도록 최적화된" 경우 `reward_hacking_risk ≈ 1.0`을 부여합니다.

집행 방식이 아름답습니다 (`evaluation/pipeline.py`의 `_build_result`):

```python
# 모든 승인이 통과하는 단 하나의 관문:
if final_approved and stage2.reward_hacking_risk >= 0.7:   # REWARD_HACKING_VETO_THRESHOLD
    final_approved = False
```

- **단일 거부권 지점**: Stage 2 통과든 Stage 3 합의든, 모든 승인 경로가 이 한 곳을 지나므로 어떤 분기도 "고신뢰 게이밍 신호를 세탁"할 수 없습니다.
- **비대칭**: 거부권은 승인→거부로만 작동하고, 이미 거부된 것을 구제하지 않습니다.
- **높은 임계값(0.7)**: "의도적으로 높다 … *고신뢰* 게이밍만 거부한다." 거부권은 강력하므로 확실할 때만 씁니다.

평가자 자신도 정직해야 합니다 — "Evaluation Transparency" 블록:

> "`questions_used`나 `evidence`가 비어 있으면 **검증 실패로 간주한다.**"

평가자가 근거 없이 점수만 내놓는 것 — 평가자 버전의 리워드 해킹 — 도 차단됩니다.

## 3.4 배심원 독립성: 같은 회사 모델은 한 표

다중 모델 합의에서 미묘한 함정: 실행한 모델과 같은 벤더의 모델이 평가하면 공정한가? `evaluation/reviewer_independence.py`는 **실행자의 벤더를 배심원단에서 제외**하는 순수 함수 정책입니다. 네 가지 정직한 라벨:

| 라벨 | 의미 |
| :--- | :--- |
| `independent` | 다른 벤더임이 **입증된** 배심원이 1명 이상 |
| `same_vendor` | 알려진 배심원 전원이 실행자와 같은 벤더 |
| `unverified` | 벤더를 알 수 없는 배심원 존재 — 입증 불가 |
| `unavailable` | 애초에 벤더 2개 미만 — 정책 적용 불가 |

핵심 규칙 두 가지:

1. **독립성은 적극적으로 입증되어야 한다** — 벤더 미상은 "다르다"로 카운트되지 않습니다(fail-closed). 하지만 같은 벤더로 *입증*되지도 않았으므로 제거되지도 않습니다.
2. **표가 순수성보다 우선** (`_MIN_VIABLE_VOTERS = 2`) — 필터링으로 배심원이 2명 미만이 되면 필터링을 포기합니다. 완벽하게 독립적인 1인 배심원보다 불완전한 2인 배심원이 낫습니다.

단일 모델밖에 없을 때는 같은 모델에 세 개의 다른 시스템 프롬프트(advocate/devil/judge)를 주되, **정직하게 `independence: UNAVAILABLE`로 라벨링**합니다. 할 수 없는 것을 하는 척하지 않습니다.

## 3.5 적대적 프로브: "완료" 주장 두드려보기

`evaluation/adversarial.py`는 9개의 명명된 적대적 클래스를 타입 레지스트리로 관리합니다: malformed_input, prompt_injection, cancel_resume, stale_state, dirty_worktree, hung_command, flaky_test, misleading_output, repeated_interrupt — 값싼 구조 검사에서 행동 검사 순으로 정렬되어 있습니다.

QA 판사(`agents/qa-judge.md`)의 규율:

> "'완료' 주장은 증거가 해당 프로브들을 견뎌냄을 보여줄 때만 강한 주장으로 인정된다."
> "당신은 제공된 증거로만 판단한다 — **직접 실행하지 않는다.**"
> "다섯 개의 구체적 차이가 스무 개의 모호한 차이를 이긴다."

실행할 수 없었던 적용 가능한 프로브는 "실행한 척"이 아니라 **증거 공백(evidence gap)** 으로 보고해야 합니다. 산출물 종류도 구분합니다: 문서 산출물에 "실행 불가"는 결코 결함이 아닙니다(완전성 렌즈로 평가).

## 3.6 일반화: 원칙 II

**완료는 승인이 아니다. 검증은 별도의 단계, 별도의 시각, 가능하면 별도의 주체가 수행한다.**

1. **세 질문 분리**: 끝났는가 / 맞는가 / 표류했는가 — 상태 타입부터 분리하라.
2. **비용 사다리**: 기계적($0) → 의미론적($$) → 합의($$$). 싼 검사가 비싼 검사를 게이팅.
3. **산출물을 평가하라, 주장을 평가하지 마라**: 에이전트의 요약이 아닌 실제 코드에 근거.
4. **게이밍을 명시적으로 탐지하라**: 하드코딩·테스트 전용 분기·자리표시자. 거부권은 단일 관문에서, 높은 임계값으로, 비대칭으로.
5. **평가자에게도 증거를 요구하라**: 근거 없는 점수는 검증 실패.
6. **검증 주체의 독립성**: 실행한 자가 승인하지 않는다. 독립성은 입증돼야 하며, 입증 못 하면 정직하게 라벨링.
7. **AI 제안 + 결정론적 게이트**: AI가 제안한 검증 명령조차 허용목록과 실존 검사를 통과해야 실행.

## 3.7 Claude Code 적용

### 작성자와 검토자 분리

같은 컨텍스트에서 코드를 쓴 에이전트에게 "검토해줘"라고 하면, 자기 결정을 정당화한 그 컨텍스트로 검토합니다. **서브에이전트로 분리하세요**:

Claude Code의 Task 도구(서브에이전트)는 독립된 컨텍스트에서 실행되므로 자연스러운 "배심원 독립성"을 제공합니다. 커스텀 에이전트(`.claude/agents/code-reviewer.md`)를 만들어 두면 더 좋습니다.

### 비용 사다리를 워크플로에

```markdown
# CLAUDE.md
## 검증 사다리
1. [$0] 코드 변경 후 항상: lint + typecheck + 관련 테스트 실행. 실패 시 여기서 멈추고 수정.
2. [$$] 기계 검사 통과 후: SPEC.md의 수용 기준과 대조해 자체 평가 (근거 인용 필수).
3. [$$$] 다음 경우에만 별도 리뷰 서브에이전트 소집: 스펙 해석이 갈린 경우,
```

hooks로 Stage 1을 강제할 수도 있습니다 — `PostToolUse` 훅(Write/Edit 매처)으로 편집 시마다 린트를 자동 실행하면, 기계 검사는 에이전트의 선의에 의존하지 않게 됩니다.

### 리워드 해킹 셀프 체크

특히 "테스트를 수정해서 통과시켰습니다"는 반드시 사람이 검토해야 하는 신호입니다. CLAUDE.md에 "실패하는 테스트는 원칙적으로 코드를 고쳐 통과시킨다. 테스트 수정이 옳다고 판단되면 반드시 이유를 먼저 보고한다"를 넣어두세요.

# 4장. 원칙 III: 기록되지 않은 것은 일어나지 않은 것이다

> "모든 상태 변화는 이벤트다; 아무것도 잃어버리지 않는다."
> (*"Event-Sourced — Every state change is an event; nothing is lost."*)
> — `docs/architecture.md`, 다섯 설계 원칙 중 #5

## 4.1 비결정성과 싸우는 유일한 방법

AI 에이전트의 실행은 본질적으로 비결정적입니다. 같은 프롬프트가 다른 결과를 내고, 긴 세션은 어디서든 끊길 수 있으며, "왜 이렇게 됐지?"라는 질문에 답할 근거가 컨텍스트와 함께 증발합니다.

Ouroboros의 대응은 이벤트 소싱(event sourcing)입니다. 그리고 이것은 기술 선택 이상의 **인식론적 입장**입니다: 시스템에 대한 모든 주장은 이벤트 로그로 뒷받침되거나, 주장이 아닙니다.

구현의 뼈대 (`events/`, `persistence/`):

- **모든 상태 변화는 불변 이벤트** — frozen Pydantic `BaseEvent`가 SQLite의 append-only 테이블에 기록됩니다. "이벤트는 결코 수정되거나 삭제될 수 없다."
- **상태는 전부 재구성** — 계보(lineage), 컨덕터 결정, 워크플로 수명주기 등 모든 읽기 모델은 이벤트 스트림을 fold하여 **투영(projection)** 됩니다. 직접 저장되는 권위 있는 상태는 없습니다.
- **단계 경계의 원자성** — `UnitOfWork`가 이벤트와 체크포인트를 모아 phase 경계에서 원자적으로 커밋합니다.
- **재현이 곧 재개** — `evolve_step()`은 이벤트 재생으로 마지막 세대의 Seed를 재구성해 **세대 중간에서** 이어갑니다. 머신이 재시작돼도 "뱀은 멈춘 자리에서 다시 시작"합니다.

## 4.2 과거형으로 명명하라

디테일이지만 깊은 디테일: 이벤트 이름은 `dot.notation.past_tense`입니다 — `orchestrator.session.started`, `execution.ac.completed`, `lineage.created`.

과거형은 문법이 아니라 **의미론**입니다. 이벤트는 "무언가를 하라"는 명령(command)이 아니라 "무언가가 이미 일어났다"는 **불변의 사실 기록**입니다. 이름부터 그렇게 지으면 이벤트를 명령으로 오용하는 설계 실수가 어색해집니다. 버전 관리도 절제되어 있습니다: `event_version`은 페이로드 안에 살아서 스키마 마이그레이션 없이 추가적(additive)으로만 진화하고, "한 버전에서 문서화된 필드는 그 버전 내에서 제거되거나 개명되지 않는다"고 약속합니다.

## 4.3 제어 결정도 기록이다: Agent OS의 잠긴 어휘

Ouroboros는 커널 용어를 **의도적으로 잠갔습니다**(`docs/contributing/agent-os-kernel-terminology.md`):

> "`AgentRuntimeContext`는 프로세스 실행 봉투, `ControlPlane`은 커널 제어 계층, `ControlContract`는 시스콜 계약, `Directive`는 명령 어휘, `ControlBus`는 하나의 로컬 전달 메커니즘, `IOJournal`은 재생 가능한 블랙박스다."

왜 용어를 잠그는가? "리뷰 코멘트와 연쇄 PR들이 런타임 컨텍스트, 제어 결정, 전송, 관측에 대해 **같은 단어**를 쓰게 하기 위해서." 동의어는 버그 취급됩니다: "기존 directive가 의미를 담고 있다면 국소적 동의어를 도입하지 마라."

이 어휘 위의 불변식들은 기록 원칙의 정수입니다:

> "종결 지시는 종결이다(Terminal directives are terminal). `CANCEL`이나 `CONVERGE` 이후 런타임은 같은 실행에 대해 새 작업을 시작해서는 안 된다."
> "모든 내구적 제어 결정은 이벤트 스토어에서 재구성 가능해야 한다. `ControlBus`를 통한 반응적 전달은 **영속화의 대체물이 아니다.**"
> "재개 시의 표류(resume drift)는 CLI 다듬기 문제가 아니라 **제어 계약 버그**로 취급하라."

`AgentProcess` 수명주기 문서의 요구사항이 이를 요약합니다:

> "재생(replay)은 다음에 답할 수 있어야 한다: 어떤 프로세스가 존재했고, 무엇을 하고 있었고, 어떤 런타임 핸들로 재개할 수 있고, **왜 멈췄는가.**"

그리고 규율: "pause/resume을 메모리에서만 되는 최선노력(best-effort)으로 만들지 마라; 상태 전이는 재생 가능해야 한다." 임시 로그나 stdout을 저널로 취급하는 것도 금지 — 블랙박스는 하나여야 합니다.

## 4.4 표류의 측정: 드리프트 공식

기록의 목적 중 하나는 **의도로부터의 거리**를 측정하는 것입니다. `observability/drift.py`:

```
combined_drift = goal_drift × 0.5 + constraint_drift × 0.3 + ontology_drift × 0.2
허용 기준: combined_drift ≤ 0.3  (DRIFT_THRESHOLD)
```

각 성분의 계산이 놀랍도록 단순합니다:

- **goal_drift** = `1 − Jaccard(목표의 단어 집합, 출력의 단어 집합)` — 단어 집합 유사도의 역수
- **constraint_drift** = `min(위반 개수 × 0.1, 1.0)` — 위반당 0.1점, 1.0 상한
- **ontology_drift** = `1 − Jaccard(Seed 온톨로지 필드명, 현재 개념명)`

프런티어 모델이 아니라 **Jaccard 유사도**입니다. 교훈: **관측 지표는 싸고 결정론적이어야 상시 가동할 수 있습니다.** 정밀한 판단(LLM)은 지표가 경보를 울린 뒤에 투입해도 됩니다 — 실제로 드리프트 > 0.3은 3장의 Stage 3 합의를 부르는 트리거입니다.

### 자동 회고 (auto-retrospective)

`observability/retrospective.py`는 **3 이터레이션마다** 자동으로 회고를 실행합니다: 드리프트를 재측정하고, 성분별 임계 초과 시 구체적 권고("목표 재초점", "주요 경로 수정 권고")를 생성하며, `combined_drift > 0.3`이면 **사람의 주의가 필요함**(`HumanAttentionRequiredEvent`)을 발신합니다. 표류는 한 번에 크게가 아니라 **조금씩 누적**되므로, 주기적 자동 점검이 있어야 조기에 잡힙니다.

## 4.5 기록 전 소독

기록 원칙의 그림자 조항: **모든 것을 기록하되, 기록 전에 소독하라.** Ouroboros 코드베이스에는 "sanitize/redact" 관련 코드가 246곳 있습니다:

- `sanitize_event_data_for_persistence()` — 원시 런타임 스트림 페이로드를 저장 전에 제거
- 컨덕터 이벤트의 시크릿 자동 마스킹 패턴
- 로그에 남기기 전 제안 명령 마스킹(`_redact`)
- 설정 파일은 `.tmp`에 쓴 뒤 `os.replace`로 원자적 교체 — 마지막 정상 설정을 절대 파괴하지 않음
- DoS 방지 컨텍스트 상한: 인터뷰 입력 50,000자, 사용자 응답 10,000자, LLM 응답 100,000자에서 절단

영속 계층은 오래 살고 널리 읽힙니다. 시크릿과 무한 크기 페이로드는 기록되는 순간 부채가 됩니다.

## 4.6 일반화: 원칙 III

**기록되지 않은 것은 일어나지 않은 것이다. 재구성할 수 없는 상태는 신뢰할 수 없는 상태다.**

1. **결정과 근거를 함께 기록하라** — "무엇을 했나"만이 아니라 "왜 멈췄나"까지 재생 가능해야 한다.
2. **기록은 불변·추가 전용** — 과거를 고치지 말고 새 사실을 덧붙여라. 이름은 과거형으로.
3. **상태는 기록의 투영** — 기록과 상태가 어긋날 수 있는 구조(이중 장부)를 만들지 마라.
4. **어휘를 잠가라** — 팀(과 에이전트)이 같은 개념에 같은 단어를 쓰게 하라. 동의어는 표류의 씨앗이다.
5. **표류 지표는 싸게, 상시로** — 결정론적 근사 지표로 상시 감시하고, 경보 시에만 비싼 판단 투입.
6. **주기적 자동 회고** — N 이터레이션마다 의도와의 거리를 재측정하고 사람 호출 기준을 명시하라.
7. **기록 전 소독** — 시크릿 마스킹, 크기 상한, 원자적 쓰기.

## 4.7 Claude Code 적용

### 작업 저널을 파일로

Claude Code 세션은 컨텍스트가 압축될 수 있습니다. 중요한 결정은 **파일**이 살아남습니다:

```markdown
# CLAUDE.md
## 작업 저널 규칙
장시간 작업에서는 docs/journal/<날짜>-<작업>.md에 추가 전용으로 기록한다:
- 내린 결정 + 근거 (과거형: "X를 선택했다. 이유: ...")
- 막힌 지점과 해결 방법
- 사용자의 교정 사항 (최우선 보존)
기존 항목은 수정하지 않는다. 정정은 새 항목으로 덧붙인다.
```

새 세션에서 "저널 읽고 이어서 해줘"만으로 재개할 수 있습니다 — 이것이 개인 수준의 이벤트 소싱입니다.

### git = 나의 이벤트 스토어

- **작은 커밋, 자주**: 각 논리적 단계마다 커밋하면 재생(replay)과 되감기(rewind)가 가능해집니다. 에이전트 실험이 잘못돼도 `git reset`으로 마지막 정상 상태로 돌아갑니다.
- **커밋 메시지에 근거를**: "fix bug"가 아니라 "왜"를 담으세요. 에이전트가 쓴 코드일수록 미래의 자신(과 미래의 에이전트)을 위한 근거가 중요합니다.
- **worktree로 실험 격리**: 큰 실험은 별도 worktree에서. Ouroboros가 brownfield 탐색에 스냅샷 worktree를 쓰는 것과 같은 이유입니다.

### hooks로 자동 기록·감시

Ouroboros의 drift-monitor 훅(PostToolUse에서 Write|Edit 감시)을 본뜬 개인 버전:

```json
// .claude/settings.json - PostToolUse 훅으로 편집 이력 자동 기록
{"hooks": {"PostToolUse": [{"matcher": "Write|Edit",
  "hooks": [{"type": "command", "command": "echo \"$(date +%T) edited\" >> .journal/edits.log"}]}]}}
```

세션 종료 후 "어떤 파일이 몇 번 고쳐졌나"를 보면 spinning(같은 파일 반복 수정 — 5장)의 물증이 남습니다.

### 드리프트 셀프 체크 의식

긴 세션 중간에 주기적으로 (Ouroboros의 3-이터레이션 회고처럼):

```
지금까지의 변경을 SPEC.md와 대조해줘:
1. 목표에서 벗어난 작업이 있었나?
2. 제약을 위반한 것이 있나?
3. 스펙에 없는 새 개념/모듈을 도입했나? 했다면 왜?
셋 중 하나라도 "예"면 계속하기 전에 나에게 보고해.
```

### 어휘 잠그기

CLAUDE.md에 프로젝트 용어집을 두세요:

```markdown
## 용어 (이 단어들만 사용)
- "Job" = 백그라운드 실행 단위 (Task, Work 등 동의어 금지)
- "Verdict" = QA 평가 결과 (Result와 구분: Result는 실행 결과)
```

에이전트가 코드·주석·커밋에서 일관된 어휘를 쓰면, 검색 가능성과 리뷰 정확도가 함께 올라갑니다.

# 5장. 원칙 IV: 반복하지 말고 진화하라

> "실행과 평가가 끝나도 시스템은 멈추지 않는다 — '우리가 아직 모르는 것은 무엇인가?'를 묻고, 그 답을 다음 세대에 되먹인다."
> — `docs/guides/evolution-loop.md`

## 5.1 루프의 두 관문

에이전트에게 "될 때까지 반복해"라고 하면 두 가지 실패가 기다립니다: **너무 일찍 시작**하거나(모호한 채로 돌진) **너무 늦게 멈추거나**(무한 루프, 또는 미성숙한 조기 종료). Ouroboros는 루프의 양 끝에 수학적 관문을 세웠습니다:

> "1. Ambiguity ≤ 0.2 — 명확해지기 전에는 만들지 마라 (인터뷰 게이트).
> 2. Similarity ≥ 0.95 — 안정되기 전에는 진화를 멈추지 마라 (수렴 게이트).
> 첫 관문은 조기 실행을 막고, 둘째 관문은 조기 종료를 막는다."

수렴은 연속 세대의 온톨로지 스키마 유사도로 측정됩니다:

```
Similarity = 0.5 × 이름_겹침 + 0.3 × 타입_일치 + 0.2 × 완전_일치
```

그리고 종료 신호는 네 가지입니다: 안정(유사도 ≥ 0.95), 정체(3세대 연속 안정), 진동(주기-2 순환 감지), **하드 캡(30세대)**. 마지막 것이 중요합니다 — 아무리 정교한 수렴 판정도 안전 밸브를 대체하지 않습니다.

## 5.2 Wonder와 Reflect: 인지 노동의 분업

2세대부터 인터뷰를 대체하는 두 엔진은 **철학과 실무의 분업**입니다:

- **Wonder** (`evolution/wonder.py`, temperature 0.7) — "우리가 아직 모르는 것은?" — "철학적이다: 시스템이 *알고* 있는 것이 아니라 *가정하고* 있는 것을 식별한다."
- **Reflect** (`evolution/reflect.py`, temperature 0.5) — "스펙이 어떻게 진화해야 하는가?" — "실용적이다: 그 공백을 구체적 변경으로 바꾼다 — 새 수용 기준, 수정된 온톨로지 필드, 조여진 제약."

temperature 차이도 의도적입니다: 질문 생성(발산)은 0.7, 스펙 수정(수렴)은 0.5, 채점(2장)은 0.1. **발산은 뜨겁게, 수렴은 미지근하게, 측정은 차갑게.**

### 근거 있는 질문만: grounded elenchus

Wonder의 프롬프트 규율은 "비판을 위한 비판"을 차단합니다:

> "AC에 근거하지도 않고 실제 공백도 아닌 질문은 질문이 아니라 **기분(mood)이다. 배출하지 마라.**"

모든 질문은 자신이 무엇에 도전하는지 명명해야 합니다: **challenge**는 특정 AC 번호를 인용하며 그 AC가 틀렸다는 구체적 증거가 있을 때만 허용되고("통과한 주장을 다시 여는 유일하게 허가된 방법"), **gap**은 목표가 요구하지만 어떤 AC도 다루지 않는 것을 명명합니다. 여기에 **범위 가드**가 겹칩니다:

> "온톨로지는 언제나 불완전하다 — 그것은 정상이지 공백이 아니다."

Seed의 목표·제약 너머의 개념 제안은 금지됩니다. 무한 확장("이것도 필요하지 않을까?")은 루프의 고질병이고, 범위 가드는 그 백신입니다.

### 만족화(satisficing): 통과한 것은 다시 열지 마라

Reflect의 가장 세련된 설계는 **AC 델타 패칭**입니다. 전체 스펙을 다시 쓰는 대신 `keep / revise / add` 패치만 허용하고:

> "통과했고, 도전받지 않았고, 회귀하지 않은 AC는 **그대로(verbatim) 유지해야 한다** — 한정된 자원을 가진 합리적 행위자는 증거 없이 만족화된 결정을 재유도하지 않는다."

LLM이 규칙을 어기고 보호 대상 AC를 고치려 하면? **결정론적 백스톱**(`_apply_satisficing_backstop`)이 LLM의 제안을 받아 최종 처분합니다: 보호 인덱스는 강제 keep, 누락 인덱스는 암묵 keep, 중복·범위 밖 패치는 폐기. **LLM은 제안하고, 결정론적 코드가 처분한다** — 3장의 명령 검증기와 같은 패턴이 여기서도 반복됩니다.

`remove` 연산이 없는 것도 의도적입니다: AC의 위치 기반 정체성이 회귀 감지의 기반이므로, 삭제는 정체성을 깨뜨립니다. 진화에도 보존해야 할 불변식이 있습니다.

## 5.3 제자리걸음의 네 얼굴

루프가 도는 것과 나아가는 것은 다릅니다. `resilience/stagnation.py`는 **정체를 네 가지 패턴으로 타입화**했습니다 — 상태 없는 순수 검출기가 실행 이력(출력 해시, 오류 시그니처, 드리프트 점수)만 보고 판정합니다:

| 패턴 | 감지 알고리즘 | 직관 |
| :--- | :--- | :--- |
| **Spinning** | 최근 3개 출력의 해시가 전부 동일 | 같은 짓을 반복하며 같은 결과 |
| **Oscillation** | 최근 4개가 A→B→A→B (짝수/홀수 인덱스 해시 비교) | 두 상태를 왕복 |
| **No-Drift** | 3회 연속 드리프트 변화량 < 0.01 | 움직이는데 위치가 그대로 |
| **Diminishing Returns** | 4회 연속 개선폭 < 0.01 | 나아지고는 있는데 기울기가 0 |

전부 SHA-256 해시와 뺄셈입니다. LLM 없음. **정체 감지가 정체되면 안 되므로**, 감지기는 싸고 결정론적이며 "절대 실패하지 않고, 감지하거나 안 하거나"입니다.

여러분 자신의 에이전트 세션에서도 이 네 패턴을 그대로 관찰할 수 있습니다: 같은 에러에 같은 수정을 반복(spinning), 두 접근을 번갈아 시도(oscillation), 활발하게 일하는데 통과 테스트 수가 그대로(no-drift), 수정할수록 개선폭이 줄어듦(diminishing returns).

## 5.4 막히면 세게가 아니라 다르게: 수평 사고

정체가 감지되면 무엇을 하는가? Ouroboros 설계 원칙 #4:

> "**수직보다 수평(Lateral Over Vertical)** — 막히면 더 세게 시도하지 말고 관점을 바꿔라."

`resilience/lateral.py`는 5개 페르소나를 정체 패턴에 **매핑**합니다:

| 정체 패턴 | 처방 페르소나 | 페르소나의 신조 |
| :--- | :--- | :--- |
| Spinning | **Hacker** | "규칙은 멈춰 설 벽이 아니라 우회할 장애물이다" — 제약 자체를 의심 |
| No-Drift | **Researcher** | "추측을 멈추고 조사를 시작하라" — 코드 대신 증거 수집 |
| Diminishing Returns | **Simplifier** | "복잡성은 밥값을 못 하면 잘린다" — 구성요소 50% 이상 제거 목표 |
| Oscillation | **Architect** | "아키텍처와 싸우고 있다면 아키텍처가 잘못됐다" — 구조적 원인 |
| (만능 대체) | **Contrarian** | "아무것도 안 하면 어떻게 되나?" — 가정 자체를 반전 |

매핑의 논리가 정확합니다: 같은 시도의 반복(spinning)은 보이지 않는 제약 가정이 원인이므로 해커가, 진전 없는 활동(no-drift)은 정보 부족이 원인이므로 연구자가, 수확 체감은 과잉 복잡성이 원인이므로 단순화가, 두 설계 사이의 진동은 구조 결함이 원인이므로 아키텍트가 처방됩니다.

각 페르소나에는 **실용의 닻**이 박혀 있습니다: Hacker의 "목표는 동작하는 코드지 이론적 우아함이 아니다", Simplifier의 "추상화는 세 번 중복된 뒤에"(rule of three), Researcher의 "테스트를 읽어라 — 실행 가능한 문서다."

그리고 **개입 예산**: `RecoveryPlanner.max_interventions = 1` (기본값). 페르소나 개입도 한 번뿐, 소진되면 정직하게 `STAGNATED`로 종결합니다. 탈출 시도조차 무한 루프가 되면 안 됩니다.

## 5.5 일반화: 원칙 IV

**루프에는 입구 관문, 출구 관문, 정체 감지기, 관점 전환 장치, 그리고 하드 캡이 있어야 한다.**

1. **시작 기준과 종료 기준을 수치로** — "충분히 명확한가"와 "충분히 안정됐는가"를 측정 가능하게.
2. **발산과 수렴을 분리** — 질문 생성과 스펙 수정은 다른 모드(다른 온도, 다른 프롬프트).
3. **비판에는 근거를** — 통과한 결정을 다시 열려면 구체적 증거를 요구하라. 기분은 질문이 아니다.
4. **만족화** — 잘 된 것은 놔둬라. 증거 없는 재작업은 예산 낭비이자 회귀 위험.
5. **정체를 타입화하라** — spinning/oscillation/no-drift/diminishing returns. 각각 다른 처방.
6. **막히면 관점을 바꿔라** — 더 큰 모델, 더 긴 재시도가 아니라 다른 시각. 단, 개입 횟수도 예산 안에서.
7. **하드 캡은 협상 불가** — 30세대든 3시간이든, 모든 루프에 안전 밸브를.

## 5.6 Claude Code 적용

### 반복 작업의 명시적 관문

에이전트에게 반복 루프를 시킬 때 세 가지를 함께 주세요:

```
테스트가 전부 통과할 때까지 수정-실행을 반복해줘. 단:
- [종료 기준] 전체 테스트 통과 + 새 lint 오류 0
- [정체 기준] 같은 테스트가 3회 연속 같은 이유로 실패하면 멈추고 나에게
  현재 가설과 시도 이력을 보고해
- [하드 캡] 최대 10회 시도. 도달하면 가장 유망한 상태에서 멈추고 요약해
```

"될 때까지"는 지시가 아닙니다. 관문 없는 루프는 토큰 소각로입니다.

### 정체 자가 감지 규칙

```markdown
# CLAUDE.md
## 정체 프로토콜
다음 신호가 보이면 시도를 멈추고 접근을 바꾼다:
- 같은 파일을 3회 이상 같은 의도로 수정 (spinning) → 제약 가정을 의심하라
- 두 접근 사이를 왕복 (oscillation) → 구조적 문제 — 한 단계 위에서 다시 설계
- 활동은 많은데 통과 테스트 수 불변 (no-drift) → 코딩을 멈추고 조사하라
접근 전환도 2회까지. 그래도 막히면 사용자에게 상황을 정리해 보고한다.
```

### 페르소나를 서브에이전트로

막혔을 때 다섯 관점을 실제로 소환할 수 있습니다:

```
이 문제로 3번 막혔어. 서로 다른 관점의 서브에이전트 3개를 병렬로 실행해줘:
1. [Simplifier] 이 설계에서 절반을 제거해도 요구사항을 만족하는 방법은?
2. [Researcher] 코드를 고치지 말고, 실패의 근본 원인에 대한 증거만 수집해줘
3. [Contrarian] 이 "문제"를 고치지 않는 것이 정답일 가능성을 논증해줘
```

`.claude/agents/`에 페르소나 에이전트 정의를 만들어 두면 재사용할 수 있습니다. Ouroboros의 `agents/*.md`가 훌륭한 템플릿입니다 — 핵심은 각 페르소나가 **단일 입장**(single stance)을 갖고, **출력 형식**(예: "구성요소 50% 제거안")까지 지정된다는 것.

### 만족화 규칙

리팩터링·반복 개선 작업에서:

```markdown
## 만족화 규칙
- 통과한 테스트, 승인된 설계, 확정된 결정은 명시적 증거 없이 다시 열지 않는다
- 스펙 변경 제안은 "무엇이 잘못됐다는 어떤 증거" 형식으로만
```

에이전트는 "더 낫게"의 유혹에 약합니다. 이미 승인된 코드를 지나가며 리팩터링하는 것은 회귀 위험 + 리뷰 부담 + 예산 낭비의 삼중 손실입니다.

---

**다음 장에서는** 이 모든 루프와 자율성을 감싸는 마지막 원칙 — **경계** — 를 다룹니다. 자율성은 공짜가 아니며, 좋은 울타리가 좋은 에이전트를 만듭니다.

# 6장. 원칙 V: 자율성에는 경계를 세워라

> "불확실할 때는 닫힌 쪽으로 실패하라(fail closed)."
> — Ouroboros 코드베이스에 106회 등장하는 문구의 요지

## 6.1 자율성의 대가

에이전트에게 자율성을 줄수록 세 가지 위험이 커집니다: **폭주**(무한 루프·무한 비용), **월권**(요청 범위 밖 행동), **침묵 실패**(불확실한 상황에서 임의로 진행). Ouroboros는 이 셋에 각각 예산, 허용목록, fail-closed로 답합니다. 이 장은 그 방어 시설의 순회입니다.

## 6.2 워치독: 활동과 진전을 구별하라

`evolution/watchdog.py`의 `GenerationProgressWatchdog`는 세 가지 시간 초과를 **구별**합니다:

| 타임아웃 | 기본값 | 감지 대상 |
| :--- | :--- | :--- |
| `generation_safety_timeout` | — | 총 경과 시간 (절대 상한) |
| `generation_idle_timeout` | 2시간 | **활동 없음** |
| `generation_no_progress_timeout` | 4시간 | 활동은 있으나 **실질 진전 없음** |

세 번째가 정수입니다. "실질 진전(material progress)"은 실질 이벤트 집합과 워크플로 지문(fingerprint)으로 추적되어, **표면적 이벤트는 시계를 리셋하지 못합니다.** 로그를 쏟아내며 바쁘게 도는 에이전트도 실질 진전이 없으면 잡힙니다 — 5장의 no-drift 감지와 같은 통찰: **활동 ≠ 진전.**

취소 방식도 계약으로 명문화되어 있습니다: `WATCHDog_CANCELLATION_MODE = "cooperative_direct_one_stage"` — 단일 협조적 취소를 보내고, 내부 태스크가 상태를 플러시하도록 기다린 뒤, `watchdog_decision`과 `control.directive.emitted` 이벤트를 **원자적으로** 기록합니다. 강제 종료의 사다리(SIGTERM→SIGKILL)가 아니라, 상태 보존을 우선하는 한 단계 협조 취소입니다. 그리고 워치독의 타이머 상태는 별도 저장 없이 영속화된 `created_at`에서 유도됩니다 — 재시작해도 예산은 이어집니다(4장의 기록 원칙이 여기서 배당금을 지급합니다).

## 6.3 비용 라우팅: 신뢰는 벌고, 실패는 승급한다

Ouroboros 설계 원칙 #1은 "**검소함 우선(Frugal First)**"입니다. `orchestrator/model_routing.py`의 모델 티어 사다리:

```
MODEL_TIER_LADDER = ("frugal", "standard", "frontier")   # 약함 → 강함
```

규칙이 경제학 교과서처럼 깔끔합니다:

- **신뢰에 의한 강등(비용 절감)**: 분해된 자식 AC가 `decomposition_trustworthy == True`(정확히 True — fail-closed 기본값은 불신)일 때만 한 단계 싼 모델로. 신뢰는 기본값이 아니라 **증명으로 버는 것**입니다.
- **실패에 의한 승급**: 재시도마다 `escalation_notches`가 누적되어 사다리를 오릅니다. "끈질기게 실패하는 자식은 사다리 전체를 걸어 올라간다" — 실패할수록 더 강한 모델을 배정받되, 천장(frontier)에서 캡.
- **대체는 위로만**: 결정된 티어에 모델이 없으면 사다리를 **위로** 걸어 대체합니다. 결정보다 싼 모델로 조용히 대체하는 일은 없습니다.
- **사용자 고정은 존중**: 사용자가 모델을 명시하면 라우팅은 휴면합니다. 자동화는 사용자의 명시적 선택을 이기지 않습니다.

추론 노력(effort) 라우팅은 **의도된 비대칭**을 보여줍니다: 모델 티어와 달리 노력은 재시도당 한 단계만 올립니다 — "이미 재시도를 태운 어려운 AC는 더 많은 추론을 벌었다." 같은 사다리 패턴이라도 축의 성질(예산 압력의 유무)에 따라 규칙이 달라집니다.

## 6.4 Fail-Closed: 불확실하면 멈춘다

코드베이스 전반에 반복되는 가장 중요한 관용구입니다. 사례들:

- `decomposition_trustworthy`의 기본값은 **불신** — 신뢰 신호가 없으면 비용 절감 없음
- 라우터 직렬화는 필드가 이상하면(중복·구식·형식 오류) 마지막-쓰기-승리로 관대하게 받지 않고 **통째로 거부**
- 배심원 독립성(3장)은 "다른 벤더임이 입증"될 때만 인정 — 미상은 독립으로 카운트 안 됨
- 명령 탐지기(3장)는 검증 불가한 AI 제안을 실행이 아니라 **skip으로 강등**
- 정체 이벤트 팩토리는 감지되지 않은 패턴의 이벤트 생성 요청에 **예외를 던짐** — 가짜 이벤트 발행 불가
- 최근 커밋 메시지가 이 문화를 증언합니다: "fail closed when investment authority is uncertain", "Make decomposition recovery evidence-driven and fail-closed"

fail-open(불확실하면 진행)은 단기적으로 매끄럽지만, 침묵 실패를 축적합니다. fail-closed는 단기적으로 번거롭지만, 모든 실패를 **보이게** 만듭니다. 1장의 safe-but-wrong이 바로 fail-open의 산물임을 기억하세요.

## 6.5 모든 루프에 예산을

Ouroboros에서 상한 없이 도는 루프는 없습니다:

| 루프 | 예산 |
| :--- | :--- |
| 진화 세대 | 최대 30세대 (하드 캡) |
| 페르소나 개입 | `max_interventions = 1`, 소진 시 `STAGNATED` 종결 |
| 모호도 채점 재시도 | 최대 10회, 형식 오류 재시도 5회 |
| 토큰 확장 | `finish_reason == "length"`(절단)일 때**만** 2배 — 실패했다고 무조건 늘리지 않음 |
| 합의 투표 | 최소 2표 미만이면 진행 대신 **오류** |
| 승급 | frontier 천장에서 캡 |
| auto 인터뷰 | 경계 있는 라운드, 느린 호출은 `blocked`/`failed`로 전이 |

토큰 확장 규칙이 특히 교육적입니다: 응답이 잘렸다는 **구체적 증거**가 있을 때만 예산을 늘립니다. "실패했으니 더 크게"가 아니라 "이 실패의 원인이 크기라는 증거가 있으니 더 크게."

## 6.6 월권 방지: 범위와 표면의 축소

- **읽기 전용 페르소나**: Socratic Interviewer는 "절대 만들지 않는다", QA 판사는 "절대 직접 실행하지 않는다" — 역할에 필요한 최소 권한만.
- **범위 가드**: Wonder는 Seed의 목표 밖 개념 제안 금지(5장).
- **파괴적 타겟 거부**: make/gradle의 deploy·publish·push·reset 타겟은 검증 명령 후보에서 원천 배제.
- **경로 탈출 차단**: 저장소 밖을 가리키는 경로 인자는 거부.
- **DoS 상한**: 모든 외부 입력에 크기 제한(4장).

## 6.7 일반화: 원칙 V

**좋은 울타리가 좋은 에이전트를 만든다. 자율성은 경계 안에서만 신뢰할 수 있다.**

1. **활동이 아니라 진전을 감시하라** — idle 타임아웃과 no-progress 타임아웃은 다른 것이다.
2. **취소는 협조적으로, 상태를 보존하며** — 그러나 반드시 실행되게.
3. **비용은 사다리로** — 싸게 시작, 실패 시 증거 기반 승급, 신뢰 시 강등, 천장과 바닥.
4. **fail-closed 기본값** — 신뢰·독립성·유효성은 증명될 때만 인정. 불확실하면 멈추고 보고.
5. **모든 루프에 예산** — 재시도, 개입, 세대, 토큰. 소진 시의 행동(정직한 종결)까지 정의.
6. **최소 권한 역할** — 인터뷰어는 못 만들고, 판사는 못 실행하게. 역할의 표면을 줄여라.
7. **자동화는 사용자의 명시적 선택에 양보한다.**

## 6.8 Claude Code 적용

### 권한 모드를 의식적으로

Claude Code의 권한 시스템이 여러분의 1차 울타리입니다:

- 기본 모드: 파괴적 작업마다 승인 — 탐색·리뷰에 적합
- `--permission-mode acceptEdits`: 파일 편집 자동 승인 — 신뢰하는 작업 흐름
- `settings.json`의 allow/deny 목록: Ouroboros의 실행 파일 허용목록처럼, **명령 단위**로 경계 설정

```json
// .claude/settings.json — 예: 테스트/린트는 자유, 패키지 설치·푸시는 확인
{"permissions": {
  "allow": ["Bash(npm test:*)", "Bash(npm run lint:*)"],
  "deny": [],
  "ask": ["Bash(git push:*)", "Bash(npm install:*)"]}}
```

핵심은 Ouroboros의 교훈 그대로: **읽기·검증은 넓게 허용하고, 쓰기·배포·설치는 좁게.**

### PreToolUse 훅 = 나의 결정론적 게이트

AI의 판단에 맡기지 말아야 할 규칙은 훅으로 강제하세요 (3장의 "AI 제안 + 결정론적 검증기" 패턴):

```bash
# PreToolUse 훅 예시: 보호 파일 편집 차단
# .claude/hooks/protect.sh — matcher: Write|Edit
if echo "$CLAUDE_TOOL_INPUT" | grep -qE '(\.env|credentials|migrations/)'; then
  echo '{"decision": "block", "reason": "보호 경로. 사용자 승인 필요"}'; exit 0
fi
```

CLAUDE.md의 지시는 에이전트가 *따르려 노력*하는 것이고, 훅은 *구조적으로 강제*되는 것입니다. 중요한 경계일수록 프롬프트가 아니라 코드로.

### 자율 실행에는 3중 예산

백그라운드/장시간 자율 작업을 맡길 때:

```
이 마이그레이션을 백그라운드로 진행해줘.
- [시도 예산] 파일당 최대 2회 시도, 실패 파일은 건너뛰고 목록화
- [시간 예산] 30분 넘으면 중간 결과를 커밋하고 상태 보고
- [범위 예산] src/legacy/ 밖의 파일은 읽기만, 수정 금지
소진 시: 멈추고 "완료/건너뜀/미착수" 3분류로 보고. 조용한 부분 성공 금지.
```

### fail-closed를 지시로

```markdown
# CLAUDE.md
## 불확실성 프로토콜
- 스펙과 코드가 모순되면: 진행하지 말고 모순을 보고한다
- 삭제·덮어쓰기 대상이 예상과 다르면: 멈추고 발견한 것을 보여준다
- 두 해석이 가능한 요구사항: 임의로 고르지 말고, 해석 A/B와 추천을 제시한다
- 검증 수단이 없는 변경: "검증 불가"를 명시하고 수동 확인 절차를 제안한다
```

### 모델·노력 사다리를 수동으로

Ouroboros의 라우팅을 사람이 흉내 낼 수 있습니다: 기계적 편집·탐색은 빠른 모델로, 설계·디버깅 교착은 강한 모델로 승급. Claude Code에서는 `/model`로 전환하거나, 서브에이전트 정의에 `model: haiku` 같은 티어를 지정해 **작업 성격별로 비용을 배정**하세요. 원칙은 같습니다: 싸게 시작하고, 실패가 증거를 만들면 승급한다.

# 7장. Claude Code 실전 종합 — 다섯 원칙을 하나의 워크플로로

> 원칙은 도구를 통해서만 습관이 된다.

이 장은 앞의 다섯 원칙을 Claude Code의 실제 기능 지형 위에 배치합니다. 먼저 대응표, 그다음 하루의 워크플로, 마지막으로 프로젝트 초기 설정 레시피입니다.

## 7.1 원칙 → 기능 대응표

| Ouroboros의 장치 | 원칙 | Claude Code의 대응물 |
| :--- | :--- | :--- |
| Socratic Interview + 모호도 게이트 | I. 명세 | **계획 모드**, 착수 전 질문 지시(CLAUDE.md), AskUserQuestion |
| 불변 Seed | I. 명세 | SPEC.md 파일, 이슈/PR 본문, 승인된 계획 |
| 3단계 평가 파이프라인 | II. 검증 | lint/test 자동화(hooks) → 자체 평가 → **리뷰 서브에이전트** |
| 배심원 독립성 | II. 검증 | 깨끗한 컨텍스트의 서브에이전트 리뷰 (구현 대화 미공유) |
| 리워드 해킹 거부권 | II. 검증 | 테스트 수정 보고 의무, 하드코딩 셀프 체크 지시 |
| 이벤트 소싱 + IOJournal | III. 기록 | **git 커밋 이력**, 작업 저널 파일, 세션 트랜스크립트 |
| 드리프트 모니터 훅 | III. 기록 | PostToolUse 훅, 주기적 "스펙 대조" 의식 |
| Wonder/Reflect + 수렴 게이트 | IV. 진화 | 명시적 종료 기준이 있는 반복 지시, 회고 후 스펙 갱신 |
| 정체 감지 + 5 페르소나 | IV. 진화 | 정체 프로토콜(CLAUDE.md), **페르소나 서브에이전트** 병렬 소집 |
| 워치독 + 예산 + fail-closed | V. 경계 | 권한 설정(allow/ask/deny), **PreToolUse 훅**, 예산 명시 지시 |
| 스킬 레지스트리 (`ooo` 명령) | 재사용 | **skills** (SKILL.md), 슬래시 커맨드 |
| 런타임 추상화 (17개 백엔드) | 이식성 | MCP 서버 — 도구를 표준 프로토콜로 노출 |

표에서 읽어야 할 것: Claude Code는 이 원칙들을 실천할 **원자재를 전부 갖추고 있습니다.** Ouroboros가 한 일은 그 원자재를 강제력 있는 파이프라인으로 조립한 것이고, 여러분은 같은 조립을 가볍게 개인 워크플로 수준에서 할 수 있습니다.

## 7.2 하루의 워크플로: 기능 하나를 다섯 원칙으로

가상의 작업 — "결제 모듈에 환불 기능 추가" — 를 처음부터 끝까지.

### ① 명세 (원칙 I) — 10분을 쓰고 2시간을 아낀다

```
> 환불 기능을 구현하기 전에, 먼저 인터뷰 모드로:
  요구사항의 모호한 부분을 하나씩 질문해줘. 가장 불확실한 것부터.
```

에이전트의 질문에 답한 뒤:

```
> 이제 SPEC-refund.md를 작성해줘: 목표 / 제약 / 수용 기준(3~7개, 각각 검증
  명령 포함) / 비목표 / 확인 필요한 가정. 작성 후 내 승인을 기다려.
```

승인 전 셀프 체크(2장의 기준): 수용 기준이 구현 단계가 아니라 **사용자 가시적 결과**인가? 가정이 요구사항으로 둔갑하지 않았는가? 비목표가 명시됐는가?

### ② 실행 준비 (원칙 V) — 울타리 먼저

- 브랜치 생성 (기록 격리): `git checkout -b feat/refund`
- 이 작업의 예산을 지시에 포함: "테스트 실패 시 같은 원인으로 3회까지만 재시도, 이후 보고"
- 민감 경로가 있다면 훅/권한 확인: 결제 모듈이면 `ask` 목록에 배포·마이그레이션 명령이 있는지

### ③ 구현 + 상시 검증 (원칙 II·III) — 작은 단위로

```
> SPEC-refund.md의 수용 기준 1번부터. TDD로: 실패하는 테스트 먼저,
  구현, 통과 확인, 커밋. 기준 하나 끝날 때마다 커밋 메시지에 "AC-1" 태그.
```

- PostToolUse 훅이 편집마다 lint를 돌리고 있다면 Stage 1($0)은 자동입니다.
- 커밋이 AC 단위로 쌓이면 git 로그가 곧 진행 기록(이벤트 스토어)입니다.

### ④ 중간 회고 (원칙 III·IV) — 세 기준마다 한 번

```
> 지금까지를 SPEC-refund.md와 대조: 벗어난 것, 위반한 제약, 스펙에 없는
  새 개념 도입 여부. 문제없으면 계속, 있으면 보고.
```

정체 신호가 보이면(같은 테스트 3연속 실패 등) 5장의 페르소나 전환: "코딩 멈추고, 실패 원인의 증거만 수집해줘" (Researcher 모드).

### ⑤ 독립 검증 (원칙 II) — 작성자가 아닌 눈으로

```
> 코드 리뷰 서브에이전트를 실행해줘. 입력은 diff와 SPEC-refund.md만.
  관점: (1) 수용 기준 충족의 증거 (2) 하드코딩/테스트 전용 분기 (3) 결제
  도메인 엣지 케이스 (부분 환불, 중복 요청, 통화 불일치)
```

리뷰어의 지적을 구현 에이전트가 반박하게 하면(advocate/devil 구조) 판단 재료가 더 좋아집니다. 최종 판정(judge)은 여러분입니다.

### ⑥ 완료 보고 (원칙 I·II) — 증거로 끝맺기

```
> 완료 보고: 수용 기준별 검증 명령과 실제 출력 원문, 미실행 검증 명시,
  알려진 한계 1줄, 그리고 SPEC의 비목표를 침범하지 않았는지 확인.
```

"Done"이 아니라 이 형식이어야 1장의 safe-but-wrong이 걸러집니다.

## 7.3 프로젝트 초기 설정 레시피

새 프로젝트(또는 기존 프로젝트에 처음 도입)에서 30분 투자로 구축하는 기반:

### CLAUDE.md — 헌법

이 책의 원칙을 압축한 템플릿:

```markdown
# CLAUDE.md

## 착수 규칙 (원칙 I)
- 복잡한 작업은 목표/제약/성공기준을 확인하고, 추측이 남으면 질문한다
- 산출물의 종류(도구/문서/스크립트)를 착수 전에 확정한다
- 승인된 스펙은 SPEC-*.md로 고정하고, 변경은 명시적 합의로만

## 검증 규칙 (원칙 II)
- 코드 변경 후: lint + typecheck + 관련 테스트 (실패 시 진행 중단)
- 실패 테스트는 코드를 고쳐 통과시킨다; 테스트 수정은 사유 보고 후에만
- 완료 보고는 검증 명령 + 출력 원문 인용으로. 미검증은 "미검증"으로 표기

## 기록 규칙 (원칙 III)
- 논리적 단위마다 커밋, 메시지에 근거 포함
- 사용자의 교정은 최우선으로 기억하고 반복 위반하지 않는다

## 반복 규칙 (원칙 IV)
- 모든 반복 루프에 종료 기준과 시도 상한을 둔다 (기본 3회)
- 같은 실패 3연속이면 접근을 바꾼다: 가정 의심 → 증거 수집 → 단순화
- 통과·승인된 것은 증거 없이 다시 열지 않는다

## 경계 규칙 (원칙 V)
- 스펙-코드 모순, 예상 밖 삭제 대상, 이중 해석: 진행하지 말고 보고
- git push, 패키지 설치, 마이그레이션, .env 접근은 사전 확인
```

### .claude/agents/ — 페르소나 상비군

최소 둘을 추천합니다:

- **code-reviewer.md** — "diff와 스펙만 입력받아, 수용 기준 충족 증거·게이밍 신호·엣지 케이스를 지적한다. 구현 과정의 사정은 고려하지 않는다."
- **simplifier.md** — "구성요소의 50% 제거를 목표로 대안을 설계한다. YAGNI, 3회 중복 전 추상화 금지."

Ouroboros의 `src/ouroboros/agents/*.md`는 페르소나 정의의 모범 답안입니다: **단일 입장 + 명시적 금지("절대 구현하지 않는다") + 출력 형식 지정.**

### .claude/settings.json — 울타리

권한 allow/ask 목록(6장), PostToolUse 린트 훅(3장), 필요시 PreToolUse 보호 경로 훅(6장).

### skills — 반복 워크플로의 결정화

이 장의 ①~⑥ 흐름 자체를 스킬로 만들 수 있습니다. `.claude/skills/spec-first/SKILL.md`:

```markdown
---
name: spec-first
description: 명세 우선 개발 사이클. 인터뷰 → SPEC 작성 → AC 단위 TDD →
  독립 리뷰 → 증거 기반 완료 보고. 신규 기능 개발 요청 시 사용.
---
1. 요구사항 인터뷰: 모호한 것부터 하나씩 질문, 사용자가 충분하다 할 때까지
2. SPEC-<기능>.md 작성 (목표/제약/AC 3~7개+검증명령/비목표/가정) 후 승인 대기
3. AC 단위로 TDD 구현, AC마다 커밋
4. 3 AC마다 스펙 대조 회고
5. 리뷰 서브에이전트 (입력: diff + SPEC만)
6. AC별 검증 출력 원문으로 완료 보고
```

Ouroboros의 `ooo interview`, `ooo run`이 하는 일의 개인화 버전입니다. **좋은 워크플로는 프롬프트로 반복하지 말고 스킬로 결정화하세요** — 그것이 Ouroboros가 21개 스킬로 한 일입니다.

### MCP — 검증 도구의 표준화

프로젝트 고유의 검증(스테이징 배포 확인, 성능 벤치마크 등)이 있다면 MCP 서버로 노출하는 것을 고려하세요. Ouroboros가 평가 파이프라인을 MCP 도구(`ouroboros_evaluate`, `ouroboros_qa`)로 노출해 **어떤 AI CLI에서도 같은 검증**을 쓸 수 있게 한 것과 같은 이유입니다: 검증이 도구에 내장되면 에이전트가 바뀌어도 품질 기준은 남습니다.

## 7.4 규모별 적용 가이드

| 규모 | 적용 강도 |
| :--- | :--- |
| **빠른 스크립트, 일회성** | 원칙 I의 최소형만: 산출물 종류 + 성공 기준 한 줄. 나머지는 과잉 |
| **개인 프로젝트** | CLAUDE.md 헌법 + git 습관 + 완료 보고 형식 |
| **팀 프로젝트** | 위 전부 + SPEC 파일 문화 + 리뷰 서브에이전트 + 권한/훅 표준화 + 용어집 |
| **자율 실행 (백그라운드/CI)** | 위 전부 + 3중 예산 명시 + fail-closed 프로토콜 + 저널 의무화 |

자율성이 커질수록 오른쪽 열이 무거워지는 것은 우연이 아닙니다 — **감독이 줄어드는 만큼 구조가 대신 감독해야 합니다.** Ouroboros가 그 극단(완전 자율 진화 루프)에서 요구한 것이 이 책의 다섯 원칙 전부였다는 사실이, 원칙들의 필요성에 대한 가장 좋은 증명입니다.

# 8장. 패턴 카탈로그 — 체크리스트와 팁

> 이 장만 프린트해서 책상에 붙여도 됩니다.

## 8.1 다섯 원칙, 다섯 문장

1. **명세** — 명확해지기 전에는 만들지 마라. 명확함은 측정하라.
2. **검증** — 완료는 승인이 아니다. 싼 검사가 비싼 검사를 게이팅하게 하라.
3. **기록** — 기록되지 않은 것은 일어나지 않은 것이다. 결정과 근거를 남겨라.
4. **진화** — 반복하지 말고 진화하라. 루프에는 입구·출구·정체 감지·하드 캡.
5. **경계** — 자율성에는 울타리를. 불확실하면 닫힌 쪽으로 실패하라.

## 8.2 착수 전 체크리스트 (원칙 I)

- [ ] 산출물의 **종류**가 확정됐는가? (실행 도구 / 문서 / 스크립트 / 분석)
- [ ] 목표가 사용자 가시적 결과로 서술됐는가?
- [ ] 제약(건드리면 안 되는 것)이 명시됐는가?
- [ ] 수용 기준이 3~7개, 각각 **검증 명령**과 함께 있는가?
- [ ] 수용 기준이 구현 단계가 아니라 결과인가? (형제 기준의 하위 단계 = 결함)
- [ ] **비목표**가 적혀 있는가?
- [ ] 가정이 요구사항으로 둔갑하지 않았는가? (사용자가 확인한 것만 계약)
- [ ] 의도적으로 유보한 결정이 "모른다"로 정직하게 기록됐는가?

## 8.3 완료 판정 체크리스트 (원칙 II) — safe-but-wrong 필터

- [ ] 산출물이 요청과 같은 **종류**인가? (도구를 원했는데 문서가 오지 않았나)
- [ ] **새 입력으로 재실행**할 수 있는가? 그 명령을 봤는가?
- [ ] 검증이 증명한 것이 **요청된 동작**인가, 파일의 존재인가?
- [ ] 결측·미확인이 `OK`/`0`/빈 값으로 위장되지 않았는가?
- [ ] 테스트가 수정됐다면 그 사유를 들었는가?
- [ ] 하드코딩된 기대값, 테스트 전용 분기가 없는가?
- [ ] 이전에 교정했던 사항이 다시 어겨지지 않았는가?
- [ ] 보고가 "Done"이 아니라 증거(명령+출력 원문) 형식인가?

## 8.4 정체 진단표 (원칙 IV)

| 증상 | 패턴 | 처방 (페르소나) |
| :--- | :--- | :--- |
| 같은 수정을 반복, 같은 실패 | **Spinning** | 제약 가정을 의심하라 (Hacker): "그 규칙, 진짜 규칙인가?" |
| 두 접근 사이를 왕복 | **Oscillation** | 구조를 의심하라 (Architect): "아키텍처와 싸우고 있다면 아키텍처가 잘못됐다" |
| 활동은 많은데 지표 불변 | **No-Drift** | 코딩을 멈추고 조사하라 (Researcher): "추측 대신 증거" |
| 개선폭이 계속 줄어듦 | **Diminishing Returns** | 절반을 버려라 (Simplifier): "동작할 수 있는 가장 단순한 것은?" |
| 전부 막힘 | — | 반전하라 (Contrarian): "아무것도 안 하면 어떻게 되나?" |

수치 기준(Ouroboros 기본값): 같은 결과 3회 = spinning, A→B→A→B 4회 = oscillation, 변화량 < 0.01 × 3회 = no-drift. 여러분의 기준도 **미리 숫자로** 정해두세요.

## 8.6 프롬프트 조각 모음

바로 복사해 쓰는 문장들:

**인터뷰 요청**
> "구현하지 말고, 먼저 모호함이 가장 큰 부분부터 하나씩 질문해줘. 충분히 명확해지면 수용 기준 3~7개짜리 명세를 만들어 승인받고 시작해."

**독립 리뷰**
> "리뷰 서브에이전트에 diff와 SPEC.md만 줘. 구현 과정의 대화는 주지 마. 관점: 수용 기준 충족 증거, 게이밍 신호(하드코딩·테스트 전용 분기), 엣지 케이스."

**게이밍 셀프 체크**
> "테스트를 통과시키려고 하드코딩, 테스트 전용 분기, 테스트 수정을 했는지 확인하고, 했다면 각각의 정당성을 설명해줘."

**드리프트 체크**
> "지금까지의 변경을 스펙과 대조해줘: 벗어난 작업, 위반한 제약, 스펙에 없는 새 개념. 하나라도 있으면 계속하기 전에 보고해."

**정체 탈출 (관점 병렬화)**
> "서브에이전트 셋을 병렬로: ① 구성요소 절반을 제거한 설계안 ② 코드 수정 없이 실패 원인의 증거만 수집 ③ 이 문제를 고치지 않는 것이 정답일 가능성의 논증."

**증거 기반 완료 보고**
> "수용 기준별로 검증 명령과 실제 출력 원문을 인용해서 보고해줘. 실행 못 한 검증은 '미실행'으로 명시하고, 알려진 한계를 한 줄 이상 적어줘."

## 8.7 안티패턴 사전

| 안티패턴 | 왜 나쁜가 | 교정 |
| :--- | :--- | :--- |
| "알아서 잘 만들어줘" | 모호함이 실행까지 관통 → 계약 표류 | 인터뷰 먼저, 명세 승인 후 착수 |
| "될 때까지 반복해" | 관문 없는 루프 = 토큰 소각로 | 종료 기준 + 정체 기준 + 하드 캡 |
| 구현한 에이전트에게 "리뷰해줘" | 자기 결정을 정당화한 컨텍스트로 검토 | 깨끗한 컨텍스트의 서브에이전트 |
| "Done"을 그대로 수락 | 완료 ≠ 승인. safe-but-wrong 통과 | 8.3 체크리스트 |
| 통과한 코드의 "지나가는 리팩터링" | 증거 없는 재작업 = 회귀 위험 + 예산 낭비 | 만족화: 승인된 것은 열지 않는다 |
| 실패하면 무조건 더 큰 모델/토큰 | 원인 진단 없는 증액 | 증거 기반 승급 (잘렸을 때만 토큰 2배) |
| 중요 규칙을 프롬프트에만 | 프롬프트는 노력, 강제가 아님 | 훅·권한 설정으로 구조화 |
| 결측을 0/OK로 채움 | 하류 판단 전부 오염 | `unchecked` / `insufficient data` 표기 |
| 임시 로그를 기록으로 취급 | 재구성 불가 = 신뢰 불가 | 커밋·저널·스펙 파일로 단일 기록 |
| 매번 같은 워크플로를 프롬프트로 반복 | 일관성 없음, 유실됨 | 스킬/에이전트 정의로 결정화 |

## 8.8 Ouroboros가 남긴 숫자들 (참고 기준값)

여러분의 기준을 정할 때 출발점으로:

| 항목 | 값 | 의미 |
| :--- | :--- | :--- |
| 모호도 게이트 | ≤ 0.2 | 가중 명확도 80%면 착수 가능 |
| 채점 온도 | 0.1 | 측정은 차갑게 (재현성) |
| 발산/수렴 온도 | 0.7 / 0.5 | 질문은 뜨겁게, 스펙 수정은 미지근하게 |
| 수용 기준 개수 | 3~7 | 그보다 잘게 = 구현 분해(월권) |
| 의미 평가 승인선 | score ≥ 0.8 | + AC 준수 100% |
| 합의 다수결 | ≥ 2/3 | 최소 2표, 미달 시 오류 |
| 리워드 해킹 거부권 | ≥ 0.7 | 높게 — 고신뢰 게이밍만 거부 |
| 드리프트 허용선 | ≤ 0.3 | goal 0.5 + constraint 0.3 + ontology 0.2 가중 |

| 페르소나 개입 예산 | 1회 | 탈출 시도도 예산 안에서 |
| 정체 판정 | 동일 결과 3회 | spinning 기준 |

## 8.9 마지막 페이지: 뱀이 가르쳐 준 것

Ouroboros라는 이름은 장식이 아니었습니다. 자기 꼬리를 삼키는 뱀 — 출력이 입력이 되는 구조 — 는 위험한 구조입니다. 피드백 루프는 오류도 증폭시키기 때문입니다. 이 코드베이스 전체는 그 위험한 구조를 안전하게 만들기 위한 조건들의 목록이라 할 수 있습니다:

- 루프에 들어가기 전에 **명확함을 측정**하고 (원칙 I),
- 루프의 각 바퀴에서 **완료와 승인을 분리**하고 (원칙 II),
- 모든 바퀴를 **재생 가능하게 기록**하고 (원칙 III),
- 바퀴가 **진전인지 공회전인지 구별**하고 (원칙 IV),
- 루프 전체를 **예산과 울타리로 감싼다** (원칙 V).

AI 에이전트와 일하는 우리 모두가 매일 만드는 것이 바로 이런 피드백 루프입니다 — 우리의 지시가 에이전트의 출력이 되고, 그 출력이 다시 우리의 다음 지시의 입력이 되는. 그 루프를 증폭기로 만들지 소각로로 만들지는 모델의 성능이 아니라 **루프의 설계**가 결정합니다.

> "뱀은 반복하지 않는다 — 진화한다."
