# 5장. 원칙 IV: 반복하지 말고 진화하라

> "실행과 평가가 끝나도 시스템은 멈추지 않는다 — '우리가 아직 모르는 것은 무엇인가?'를 묻고, 그 답을 다음 세대에 되먹인다."
> — `docs/guides/evolution-loop.md`

## 5.1 루프의 두 관문

에이전트에게 "될 때까지 반복해"라고 하면 두 가지 실패가 기다립니다: **너무 일찍 시작**하거나(모호한 채로 돌진) **너무 늦게 멈추거나**(무한 루프, 또는 미성숙한 조기 종료). Ouroboros는 루프의 양 끝에 수학적 관문을 세웠습니다:

> "1. Ambiguity ≤ 0.2 — 명확해지기 전에는 만들지 마라 (인터뷰 게이트).
> 2. Similarity ≥ 0.95 — 안정되기 전에는 진화를 멈추지 마라 (수렴 게이트).
> 첫 관문은 조기 실행을 막고, 둘째 관문은 조기 종료를 막는다."

수렴은 연속 세대의 온톨로지 스키마 유사도로 측정됩니다:

```
Similarity = 0.5 × 이름_겹침 + 0.3 × 타입_일치 + 0.2 × 완전_일치
```

그리고 종료 신호는 네 가지입니다: 안정(유사도 ≥ 0.95), 정체(3세대 연속 안정), 진동(주기-2 순환 감지), **하드 캡(30세대)**. 마지막 것이 중요합니다 — 아무리 정교한 수렴 판정도 안전 밸브를 대체하지 않습니다.

## 5.2 Wonder와 Reflect: 인지 노동의 분업

2세대부터 인터뷰를 대체하는 두 엔진은 **철학과 실무의 분업**입니다:

- **Wonder** (`evolution/wonder.py`, temperature 0.7) — "우리가 아직 모르는 것은?" — "철학적이다: 시스템이 *알고* 있는 것이 아니라 *가정하고* 있는 것을 식별한다."
- **Reflect** (`evolution/reflect.py`, temperature 0.5) — "스펙이 어떻게 진화해야 하는가?" — "실용적이다: 그 공백을 구체적 변경으로 바꾼다 — 새 수용 기준, 수정된 온톨로지 필드, 조여진 제약."

temperature 차이도 의도적입니다: 질문 생성(발산)은 0.7, 스펙 수정(수렴)은 0.5, 채점(2장)은 0.1. **발산은 뜨겁게, 수렴은 미지근하게, 측정은 차갑게.**

### 근거 있는 질문만: grounded elenchus

Wonder의 프롬프트 규율은 "비판을 위한 비판"을 차단합니다:

> "AC에 근거하지도 않고 실제 공백도 아닌 질문은 질문이 아니라 **기분(mood)이다. 배출하지 마라.**"

모든 질문은 자신이 무엇에 도전하는지 명명해야 합니다: **challenge**는 특정 AC 번호를 인용하며 그 AC가 틀렸다는 구체적 증거가 있을 때만 허용되고("통과한 주장을 다시 여는 유일하게 허가된 방법"), **gap**은 목표가 요구하지만 어떤 AC도 다루지 않는 것을 명명합니다. 여기에 **범위 가드**가 겹칩니다:

> "온톨로지는 언제나 불완전하다 — 그것은 정상이지 공백이 아니다."

Seed의 목표·제약 너머의 개념 제안은 금지됩니다. 무한 확장("이것도 필요하지 않을까?")은 루프의 고질병이고, 범위 가드는 그 백신입니다.

### 만족화(satisficing): 통과한 것은 다시 열지 마라

Reflect의 가장 세련된 설계는 **AC 델타 패칭**입니다. 전체 스펙을 다시 쓰는 대신 `keep / revise / add` 패치만 허용하고:

> "통과했고, 도전받지 않았고, 회귀하지 않은 AC는 **그대로(verbatim) 유지해야 한다** — 한정된 자원을 가진 합리적 행위자는 증거 없이 만족화된 결정을 재유도하지 않는다."

LLM이 규칙을 어기고 보호 대상 AC를 고치려 하면? **결정론적 백스톱**(`_apply_satisficing_backstop`)이 LLM의 제안을 받아 최종 처분합니다: 보호 인덱스는 강제 keep, 누락 인덱스는 암묵 keep, 중복·범위 밖 패치는 폐기. **LLM은 제안하고, 결정론적 코드가 처분한다** — 3장의 명령 검증기와 같은 패턴이 여기서도 반복됩니다.

`remove` 연산이 없는 것도 의도적입니다: AC의 위치 기반 정체성이 회귀 감지의 기반이므로, 삭제는 정체성을 깨뜨립니다. 진화에도 보존해야 할 불변식이 있습니다.

## 5.3 제자리걸음의 네 얼굴

루프가 도는 것과 나아가는 것은 다릅니다. `resilience/stagnation.py`는 **정체를 네 가지 패턴으로 타입화**했습니다 — 상태 없는 순수 검출기가 실행 이력(출력 해시, 오류 시그니처, 드리프트 점수)만 보고 판정합니다:

| 패턴 | 감지 알고리즘 | 직관 |
| :--- | :--- | :--- |
| **Spinning** | 최근 3개 출력의 해시가 전부 동일 | 같은 짓을 반복하며 같은 결과 |
| **Oscillation** | 최근 4개가 A→B→A→B (짝수/홀수 인덱스 해시 비교) | 두 상태를 왕복 |
| **No-Drift** | 3회 연속 드리프트 변화량 < 0.01 | 움직이는데 위치가 그대로 |
| **Diminishing Returns** | 4회 연속 개선폭 < 0.01 | 나아지고는 있는데 기울기가 0 |

전부 SHA-256 해시와 뺄셈입니다. LLM 없음. **정체 감지가 정체되면 안 되므로**, 감지기는 싸고 결정론적이며 "절대 실패하지 않고, 감지하거나 안 하거나"입니다.

여러분 자신의 에이전트 세션에서도 이 네 패턴을 그대로 관찰할 수 있습니다: 같은 에러에 같은 수정을 반복(spinning), 두 접근을 번갈아 시도(oscillation), 활발하게 일하는데 통과 테스트 수가 그대로(no-drift), 수정할수록 개선폭이 줄어듦(diminishing returns).

## 5.4 막히면 세게가 아니라 다르게: 수평 사고

정체가 감지되면 무엇을 하는가? Ouroboros 설계 원칙 #4:

> "**수직보다 수평(Lateral Over Vertical)** — 막히면 더 세게 시도하지 말고 관점을 바꿔라."

`resilience/lateral.py`는 5개 페르소나를 정체 패턴에 **매핑**합니다:

| 정체 패턴 | 처방 페르소나 | 페르소나의 신조 |
| :--- | :--- | :--- |
| Spinning | **Hacker** | "규칙은 멈춰 설 벽이 아니라 우회할 장애물이다" — 제약 자체를 의심 |
| No-Drift | **Researcher** | "추측을 멈추고 조사를 시작하라" — 코드 대신 증거 수집 |
| Diminishing Returns | **Simplifier** | "복잡성은 밥값을 못 하면 잘린다" — 구성요소 50% 이상 제거 목표 |
| Oscillation | **Architect** | "아키텍처와 싸우고 있다면 아키텍처가 잘못됐다" — 구조적 원인 |
| (만능 대체) | **Contrarian** | "아무것도 안 하면 어떻게 되나?" — 가정 자체를 반전 |

매핑의 논리가 정확합니다: 같은 시도의 반복(spinning)은 보이지 않는 제약 가정이 원인이므로 해커가, 진전 없는 활동(no-drift)은 정보 부족이 원인이므로 연구자가, 수확 체감은 과잉 복잡성이 원인이므로 단순화가, 두 설계 사이의 진동은 구조 결함이 원인이므로 아키텍트가 처방됩니다.

각 페르소나에는 **실용의 닻**이 박혀 있습니다: Hacker의 "목표는 동작하는 코드지 이론적 우아함이 아니다", Simplifier의 "추상화는 세 번 중복된 뒤에"(rule of three), Researcher의 "테스트를 읽어라 — 실행 가능한 문서다."

그리고 **개입 예산**: `RecoveryPlanner.max_interventions = 1` (기본값). 페르소나 개입도 한 번뿐, 소진되면 정직하게 `STAGNATED`로 종결합니다. 탈출 시도조차 무한 루프가 되면 안 됩니다.

## 5.5 일반화: 원칙 IV

**루프에는 입구 관문, 출구 관문, 정체 감지기, 관점 전환 장치, 그리고 하드 캡이 있어야 한다.**

1. **시작 기준과 종료 기준을 수치로** — "충분히 명확한가"와 "충분히 안정됐는가"를 측정 가능하게.
2. **발산과 수렴을 분리** — 질문 생성과 스펙 수정은 다른 모드(다른 온도, 다른 프롬프트).
3. **비판에는 근거를** — 통과한 결정을 다시 열려면 구체적 증거를 요구하라. 기분은 질문이 아니다.
4. **만족화** — 잘 된 것은 놔둬라. 증거 없는 재작업은 예산 낭비이자 회귀 위험.
5. **정체를 타입화하라** — spinning/oscillation/no-drift/diminishing returns. 각각 다른 처방.
6. **막히면 관점을 바꿔라** — 더 큰 모델, 더 긴 재시도가 아니라 다른 시각. 단, 개입 횟수도 예산 안에서.
7. **하드 캡은 협상 불가** — 30세대든 3시간이든, 모든 루프에 안전 밸브를.

## 5.6 Claude Code 적용

### 반복 작업의 명시적 관문

에이전트에게 반복 루프를 시킬 때 세 가지를 함께 주세요:

```
테스트가 전부 통과할 때까지 수정-실행을 반복해줘. 단:
- [종료 기준] 전체 테스트 통과 + 새 lint 오류 0
- [정체 기준] 같은 테스트가 3회 연속 같은 이유로 실패하면 멈추고 나에게
  현재 가설과 시도 이력을 보고해
- [하드 캡] 최대 10회 시도. 도달하면 가장 유망한 상태에서 멈추고 요약해
```

"될 때까지"는 지시가 아닙니다. 관문 없는 루프는 토큰 소각로입니다.

### 정체 자가 감지 규칙

```markdown
# CLAUDE.md
## 정체 프로토콜
다음 신호가 보이면 시도를 멈추고 접근을 바꾼다:
- 같은 파일을 3회 이상 같은 의도로 수정 (spinning) → 제약 가정을 의심하라
- 두 접근 사이를 왕복 (oscillation) → 구조적 문제 — 한 단계 위에서 다시 설계
- 활동은 많은데 통과 테스트 수 불변 (no-drift) → 코딩을 멈추고 조사하라
접근 전환도 2회까지. 그래도 막히면 사용자에게 상황을 정리해 보고한다.
```

### 페르소나를 서브에이전트로

막혔을 때 다섯 관점을 실제로 소환할 수 있습니다:

```
이 문제로 3번 막혔어. 서로 다른 관점의 서브에이전트 3개를 병렬로 실행해줘:
1. [Simplifier] 이 설계에서 절반을 제거해도 요구사항을 만족하는 방법은?
2. [Researcher] 코드를 고치지 말고, 실패의 근본 원인에 대한 증거만 수집해줘
3. [Contrarian] 이 "문제"를 고치지 않는 것이 정답일 가능성을 논증해줘
```

`.claude/agents/`에 페르소나 에이전트 정의를 만들어 두면 재사용할 수 있습니다. Ouroboros의 `agents/*.md`가 훌륭한 템플릿입니다 — 핵심은 각 페르소나가 **단일 입장**(single stance)을 갖고, **출력 형식**(예: "구성요소 50% 제거안")까지 지정된다는 것.

### 만족화 규칙

리팩터링·반복 개선 작업에서:

```markdown
## 만족화 규칙
- 통과한 테스트, 승인된 설계, 확정된 결정은 명시적 증거 없이 다시 열지 않는다
- 스펙 변경 제안은 "무엇이 잘못됐다는 어떤 증거" 형식으로만
```

에이전트는 "더 낫게"의 유혹에 약합니다. 이미 승인된 코드를 지나가며 리팩터링하는 것은 회귀 위험 + 리뷰 부담 + 예산 낭비의 삼중 손실입니다.
