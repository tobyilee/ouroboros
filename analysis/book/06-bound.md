# 6장. 원칙 V: 자율성에는 경계를 세워라

> "불확실할 때는 닫힌 쪽으로 실패하라(fail closed)."
> — Ouroboros 코드베이스에 106회 등장하는 문구의 요지

## 6.1 자율성의 대가

에이전트에게 자율성을 줄수록 세 가지 위험이 커집니다: **폭주**(무한 루프·무한 비용), **월권**(요청 범위 밖 행동), **침묵 실패**(불확실한 상황에서 임의로 진행). Ouroboros는 이 셋에 각각 예산, 허용목록, fail-closed로 답합니다. 이 장은 그 방어 시설의 순회입니다.

## 6.2 워치독: 활동과 진전을 구별하라

`evolution/watchdog.py`의 `GenerationProgressWatchdog`는 세 가지 시간 초과를 **구별**합니다:

| 타임아웃 | 기본값 | 감지 대상 |
| :--- | :--- | :--- |
| `generation_safety_timeout` | — | 총 경과 시간 (절대 상한) |
| `generation_idle_timeout` | 2시간 | **활동 없음** |
| `generation_no_progress_timeout` | 4시간 | 활동은 있으나 **실질 진전 없음** |

세 번째가 정수입니다. "실질 진전(material progress)"은 실질 이벤트 집합과 워크플로 지문(fingerprint)으로 추적되어, **표면적 이벤트는 시계를 리셋하지 못합니다.** 로그를 쏟아내며 바쁘게 도는 에이전트도 실질 진전이 없으면 잡힙니다 — 5장의 no-drift 감지와 같은 통찰: **활동 ≠ 진전.**

취소 방식도 계약으로 명문화되어 있습니다: `WATCHDog_CANCELLATION_MODE = "cooperative_direct_one_stage"` — 단일 협조적 취소를 보내고, 내부 태스크가 상태를 플러시하도록 기다린 뒤, `watchdog_decision`과 `control.directive.emitted` 이벤트를 **원자적으로** 기록합니다. 강제 종료의 사다리(SIGTERM→SIGKILL)가 아니라, 상태 보존을 우선하는 한 단계 협조 취소입니다. 그리고 워치독의 타이머 상태는 별도 저장 없이 영속화된 `created_at`에서 유도됩니다 — 재시작해도 예산은 이어집니다(4장의 기록 원칙이 여기서 배당금을 지급합니다).

## 6.3 비용 라우팅: 신뢰는 벌고, 실패는 승급한다

Ouroboros 설계 원칙 #1은 "**검소함 우선(Frugal First)**"입니다. `orchestrator/model_routing.py`의 모델 티어 사다리:

```
MODEL_TIER_LADDER = ("frugal", "standard", "frontier")   # 약함 → 강함
```

규칙이 경제학 교과서처럼 깔끔합니다:

- **신뢰에 의한 강등(비용 절감)**: 분해된 자식 AC가 `decomposition_trustworthy == True`(정확히 True — fail-closed 기본값은 불신)일 때만 한 단계 싼 모델로. 신뢰는 기본값이 아니라 **증명으로 버는 것**입니다.
- **실패에 의한 승급**: 재시도마다 `escalation_notches`가 누적되어 사다리를 오릅니다. "끈질기게 실패하는 자식은 사다리 전체를 걸어 올라간다" — 실패할수록 더 강한 모델을 배정받되, 천장(frontier)에서 캡.
- **대체는 위로만**: 결정된 티어에 모델이 없으면 사다리를 **위로** 걸어 대체합니다. 결정보다 싼 모델로 조용히 대체하는 일은 없습니다.
- **사용자 고정은 존중**: 사용자가 모델을 명시하면 라우팅은 휴면합니다. 자동화는 사용자의 명시적 선택을 이기지 않습니다.

추론 노력(effort) 라우팅은 **의도된 비대칭**을 보여줍니다: 모델 티어와 달리 노력은 재시도당 한 단계만 올립니다 — "이미 재시도를 태운 어려운 AC는 더 많은 추론을 벌었다." 같은 사다리 패턴이라도 축의 성질(예산 압력의 유무)에 따라 규칙이 달라집니다.

## 6.4 Fail-Closed: 불확실하면 멈춘다

코드베이스 전반에 반복되는 가장 중요한 관용구입니다. 사례들:

- `decomposition_trustworthy`의 기본값은 **불신** — 신뢰 신호가 없으면 비용 절감 없음
- 라우터 직렬화는 필드가 이상하면(중복·구식·형식 오류) 마지막-쓰기-승리로 관대하게 받지 않고 **통째로 거부**
- 배심원 독립성(3장)은 "다른 벤더임이 입증"될 때만 인정 — 미상은 독립으로 카운트 안 됨
- 명령 탐지기(3장)는 검증 불가한 AI 제안을 실행이 아니라 **skip으로 강등**
- 정체 이벤트 팩토리는 감지되지 않은 패턴의 이벤트 생성 요청에 **예외를 던짐** — 가짜 이벤트 발행 불가
- 최근 커밋 메시지가 이 문화를 증언합니다: "fail closed when investment authority is uncertain", "Make decomposition recovery evidence-driven and fail-closed"

fail-open(불확실하면 진행)은 단기적으로 매끄럽지만, 침묵 실패를 축적합니다. fail-closed는 단기적으로 번거롭지만, 모든 실패를 **보이게** 만듭니다. 1장의 safe-but-wrong이 바로 fail-open의 산물임을 기억하세요.

## 6.5 모든 루프에 예산을

Ouroboros에서 상한 없이 도는 루프는 없습니다:

| 루프 | 예산 |
| :--- | :--- |
| 진화 세대 | 최대 30세대 (하드 캡) |
| 페르소나 개입 | `max_interventions = 1`, 소진 시 `STAGNATED` 종결 |
| 모호도 채점 재시도 | 최대 10회, 형식 오류 재시도 5회 |
| 토큰 확장 | `finish_reason == "length"`(절단)일 때**만** 2배 — 실패했다고 무조건 늘리지 않음 |
| 합의 투표 | 최소 2표 미만이면 진행 대신 **오류** |
| 승급 | frontier 천장에서 캡 |
| auto 인터뷰 | 경계 있는 라운드, 느린 호출은 `blocked`/`failed`로 전이 |

토큰 확장 규칙이 특히 교육적입니다: 응답이 잘렸다는 **구체적 증거**가 있을 때만 예산을 늘립니다. "실패했으니 더 크게"가 아니라 "이 실패의 원인이 크기라는 증거가 있으니 더 크게."

## 6.6 월권 방지: 범위와 표면의 축소

- **읽기 전용 페르소나**: Socratic Interviewer는 "절대 만들지 않는다", QA 판사는 "절대 직접 실행하지 않는다" — 역할에 필요한 최소 권한만.
- **범위 가드**: Wonder는 Seed의 목표 밖 개념 제안 금지(5장).
- **파괴적 타겟 거부**: make/gradle의 deploy·publish·push·reset 타겟은 검증 명령 후보에서 원천 배제.
- **경로 탈출 차단**: 저장소 밖을 가리키는 경로 인자는 거부.
- **DoS 상한**: 모든 외부 입력에 크기 제한(4장).

## 6.7 일반화: 원칙 V

**좋은 울타리가 좋은 에이전트를 만든다. 자율성은 경계 안에서만 신뢰할 수 있다.**

1. **활동이 아니라 진전을 감시하라** — idle 타임아웃과 no-progress 타임아웃은 다른 것이다.
2. **취소는 협조적으로, 상태를 보존하며** — 그러나 반드시 실행되게.
3. **비용은 사다리로** — 싸게 시작, 실패 시 증거 기반 승급, 신뢰 시 강등, 천장과 바닥.
4. **fail-closed 기본값** — 신뢰·독립성·유효성은 증명될 때만 인정. 불확실하면 멈추고 보고.
5. **모든 루프에 예산** — 재시도, 개입, 세대, 토큰. 소진 시의 행동(정직한 종결)까지 정의.
6. **최소 권한 역할** — 인터뷰어는 못 만들고, 판사는 못 실행하게. 역할의 표면을 줄여라.
7. **자동화는 사용자의 명시적 선택에 양보한다.**

## 6.8 Claude Code 적용

### 권한 모드를 의식적으로

Claude Code의 권한 시스템이 여러분의 1차 울타리입니다:

- 기본 모드: 파괴적 작업마다 승인 — 탐색·리뷰에 적합
- `--permission-mode acceptEdits`: 파일 편집 자동 승인 — 신뢰하는 작업 흐름
- `settings.json`의 allow/deny 목록: Ouroboros의 실행 파일 허용목록처럼, **명령 단위**로 경계 설정

```json
// .claude/settings.json — 예: 테스트/린트는 자유, 패키지 설치·푸시는 확인
{"permissions": {
  "allow": ["Bash(npm test:*)", "Bash(npm run lint:*)"],
  "deny": [],
  "ask": ["Bash(git push:*)", "Bash(npm install:*)"]}}
```

핵심은 Ouroboros의 교훈 그대로: **읽기·검증은 넓게 허용하고, 쓰기·배포·설치는 좁게.**

### PreToolUse 훅 = 나의 결정론적 게이트

AI의 판단에 맡기지 말아야 할 규칙은 훅으로 강제하세요 (3장의 "AI 제안 + 결정론적 검증기" 패턴):

```bash
# PreToolUse 훅 예시: 보호 파일 편집 차단
# .claude/hooks/protect.sh — matcher: Write|Edit
if echo "$CLAUDE_TOOL_INPUT" | grep -qE '(\.env|credentials|migrations/)'; then
  echo '{"decision": "block", "reason": "보호 경로. 사용자 승인 필요"}'; exit 0
fi
```

CLAUDE.md의 지시는 에이전트가 *따르려 노력*하는 것이고, 훅은 *구조적으로 강제*되는 것입니다. 중요한 경계일수록 프롬프트가 아니라 코드로.

### 자율 실행에는 3중 예산

백그라운드/장시간 자율 작업을 맡길 때:

```
이 마이그레이션을 백그라운드로 진행해줘.
- [시도 예산] 파일당 최대 2회 시도, 실패 파일은 건너뛰고 목록화
- [시간 예산] 30분 넘으면 중간 결과를 커밋하고 상태 보고
- [범위 예산] src/legacy/ 밖의 파일은 읽기만, 수정 금지
소진 시: 멈추고 "완료/건너뜀/미착수" 3분류로 보고. 조용한 부분 성공 금지.
```

### fail-closed를 지시로

```markdown
# CLAUDE.md
## 불확실성 프로토콜
- 스펙과 코드가 모순되면: 진행하지 말고 모순을 보고한다
- 삭제·덮어쓰기 대상이 예상과 다르면: 멈추고 발견한 것을 보여준다
- 두 해석이 가능한 요구사항: 임의로 고르지 말고, 해석 A/B와 추천을 제시한다
- 검증 수단이 없는 변경: "검증 불가"를 명시하고 수동 확인 절차를 제안한다
```

### 모델·노력 사다리를 수동으로

Ouroboros의 라우팅을 사람이 흉내 낼 수 있습니다: 기계적 편집·탐색은 빠른 모델로, 설계·디버깅 교착은 강한 모델로 승급. Claude Code에서는 `/model`로 전환하거나, 서브에이전트 정의에 `model: haiku` 같은 티어를 지정해 **작업 성격별로 비용을 배정**하세요. 원칙은 같습니다: 싸게 시작하고, 실패가 증거를 만들면 승급한다.
