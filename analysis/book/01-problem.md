# 1장. 입력의 실패 — Safe-but-Wrong의 세계

> "대부분의 AI 코딩 실패는 출력이 아니라 **입력**에서 발생한다. 병목은 AI의 능력이 아니라 인간의 명확성이다."
> — Ouroboros README

## 1.1 우리가 겪는 실패의 진짜 모양

AI 에이전트에게 일을 시켜본 사람이라면 누구나 이런 경험이 있습니다:

- "태스크 관리 앱 만들어줘" → 그럴듯한 코드가 나왔지만 내가 원한 것이 아니다
- 에이전트가 "완료했습니다!"라고 했지만 실행해 보니 절반만 동작한다
- 세 시간짜리 자율 실행이 끝났는데, 처음 요구사항에서 슬금슬금 벗어나 있다
- 테스트는 전부 통과하는데... 테스트가 검증하는 것이 내 요구사항이 아니다

이 실패들의 공통점은 무엇일까요? **에이전트가 멍청해서가 아닙니다.** 에이전트는 주어진 입력에 대해 대체로 합리적으로 행동했습니다. 문제는 입력 — 즉 우리가 전달한 의도 — 이 애초에 모호했고, 그 모호함이 어디서도 **측정되거나 차단되지 않은 채** 실행 단계까지 흘러들어 갔다는 것입니다.

## 1.2 가장 위험한 실패: Safe-but-Wrong

Ouroboros 문서 중 가장 통찰력 있는 글은 `docs/guides/safe-but-wrong-output.md`입니다. 이 문서는 에이전트 실패 중 **가장 잡기 어려운 부류**에 이름을 붙입니다:

> "실행 결과는 안전하고, 파괴적이지 않고, 심지어 유용하면서도, 사용자가 말한 목표에 대해서는 **틀릴** 수 있다."
> (*"can be safe, non-destructive, and even useful, while still being wrong for the user's stated goal."*)

핵심 속성은 이것입니다:

> "일반적인 안전 검사는 전부 통과할 수 있다. 파일은 삭제되지 않았고, 외부 시스템은 변경되지 않았고, 테스트는 통과할 수 있고, 출력은 읽을 만하다. 그런데도 틀렸다 — **산출물 계약(artifact contract)이 표류했기 때문이다.**"

그리고 왜 이것이 눈에 보이는 실패보다 나쁜지:

> "명백한 차단(blocker)보다 나쁘다. **거짓 확신(false confidence)을 만들기 때문이다.**"

### Safe-but-Wrong의 분류학

Ouroboros가 문서화한 실제 사례 유형들 — 여러분도 분명 겪어봤을 것입니다:

| 사용자가 요청한 것 | 에이전트가 만든 것 |
| :--- | :--- |
| 실행 가능한 도구 | 인수인계 문서 뭉치 |
| 재사용 가능한 워크플로 | 일회성 체크리스트 |
| 배치 처리 표면 | 단일 케이스 요약 |
| 결측 데이터의 정직한 표기 | `OK`, `0`, 빈 값으로 위장된 결측치 |

마지막 항목이 특히 교활합니다. "데이터가 없어서 확인 못 함"을 `0`이나 `OK`로 표기하면, 하류의 모든 판단이 오염됩니다. Ouroboros는 이를 명시적으로 금지합니다: 결측 입력은 `insufficient data` / `unchecked`로 표기해야 합니다.

### 보고 언어의 대비

같은 상황에 대한 두 가지 보고:

```
나쁨:  "Done. Created the handoff package."
좋음:  "Status: partial/supporting-output, not complete."
```

첫 번째는 실제로 한 일(문서 작성)을 완료 선언으로 포장합니다. 두 번째는 산출물의 **종류**가 요청과 다르다는 사실을 정직하게 드러냅니다. Agentic Coding에서 보고 언어의 정확성은 사치가 아니라 안전장치입니다 — 에이전트의 보고가 다음 판단의 입력이 되기 때문입니다.

## 1.3 방어: 산출물 계약 6점 검사

Ouroboros는 safe-but-wrong을 잡기 위해 산출물을 원래 계약과 대조하는 6가지 질문을 제시합니다:

1. **산출물의 종류(class)** — 사용자가 요청한 것과 같은 *종류*의 것인가? (도구 vs 문서)
2. **실행 표면** — "새 입력을 넣어 다시 실행할 수 있는가?"
3. **보조 산출물의 위장** — 지원 자료가 최종 산출물 행세를 하고 있지 않은가?
4. **결측 데이터 의미론** — 없는 것이 없다고 표기되어 있는가?
5. **과거 교정 이력** — 사용자가 이전에 바로잡은 사항이 다시 어겨지지 않았는가?
6. **검증 증거** — "검증이 *요청된 동작*을 증명했는가, 아니면 *어떤 파일이 존재한다는 것만* 증명했는가?"

6번이 백미입니다. "빌드가 성공했다"와 "요청한 기능이 동작한다"는 서로 다른 명제인데, 에이전트의 자기 보고에서는 자주 뒤섞입니다.

균형 감각도 중요합니다. 이 문서는 모든 임의 판단을 금지하지 않습니다:

> "보수적 기본값은 국소적이고 되돌릴 수 있는 공백을 메울 때는 괜찮다. **사용자가 받기로 한 것의 종류를 바꿀 때** 위험해진다."

에이전트가 사소한 결정(변수명, 파일 배치)을 스스로 내리는 것은 생산적입니다. 위험한 것은 산출물의 **본질**을 바꾸는 무단 결정입니다.

## 1.4 일반화: Agentic Coding의 제1 문제 설정

이 장에서 추출할 원칙:

**원칙 0 (문제 인식): 에이전트 실패의 기본형은 "능력 부족"이 아니라 "계약 표류"다.**

따라서 Agentic Coding 체계가 가장 먼저 갖춰야 할 것은 더 강한 모델이 아니라:

1. **계약을 명시하는 장치** — 무엇을 만들기로 했는지가 검사 가능한 형태로 존재해야 한다 (→ 2장)
2. **계약과 대조하는 장치** — "완료" 주장을 계약과 대조 검증해야 한다 (→ 3장)
3. **표류를 측정하는 장치** — 진행 중에 계약에서 얼마나 멀어졌는지 관측해야 한다 (→ 4장)

## 1.5 Claude Code 적용

이 장의 통찰을 오늘 바로 적용하는 방법:

### 산출물 종류를 먼저 못 박기

작업 요청 시 산출물의 **종류**를 명시하세요. Claude Code의 CLAUDE.md나 프롬프트에:

```markdown
## 산출물 계약
- 이 작업의 최종 산출물: 실행 가능한 CLI 도구 (문서 아님)
- 완료 기준: `mytool --input sample.csv`가 새 입력에 대해 재실행 가능해야 함
- 결측/미확인 항목은 반드시 "unchecked"로 표기 (OK/0으로 채우지 말 것)
```

### "Done" 검증 습관

에이전트가 완료를 선언하면 6점 검사 중 최소 2개를 물어보세요:

- "새 입력으로 다시 실행하는 명령어를 보여줘" (실행 표면)
- "검증이 증명한 것이 정확히 뭐야? 파일 존재? 아니면 요청한 동작?" (검증 증거)

### 보고 언어 규칙을 CLAUDE.md에

```markdown
## 보고 규칙
- 부분 완료는 "부분 완료"로 보고한다. 지원 산출물을 완료로 포장하지 않는다.
- 테스트 실패, 건너뛴 단계는 숨기지 말고 그대로 보고한다.
```

Claude Code는 CLAUDE.md의 지시를 시스템 프롬프트 수준으로 준수하므로, 보고 정직성 규칙은 여기에 두는 것이 가장 효과적입니다.
